    07 | Prompt Engineering 与 AI 编程工具¶

    面试手册 1.5 节补充

    一、核心概念速记¶

    1. Prompt Engineering 是什么？¶

    一句话：Prompt Engineering = 通过设计更好的输入（提示词），让 LLM 输出更精准、更符合预期的答案。

    为什么重要： - LLM 的输出质量 80% 取决于输入 Prompt 的质量 - 同样的模型，好的 Prompt 和差的 Prompt 输出天差地别 - 面试中常问"怎么设计一个好的系统提示词"

    2. Prompt 设计原则¶

    原则 说明 示例
    角色设定 给 LLM 一个明确的身份 "你是一位资深 Python 工程师"
    任务明确 说清楚要做什么 "请解释下面代码的作用，并指出潜在问题"
    输出格式 规定输出结构 "请用 JSON 格式返回，包含 'summary' 和 'issues' 两个字段"
    Few-Shot 给几个示例 "示例1：输入 X -> 输出 Y；示例2：输入 A -> 输出 B"
    思维链 CoT 让 LLM 一步步思考 "请先分析需求，再给出方案，最后总结"
    边界约束 说清不能做什么 "如果信息不足，请明确说明无法回答，不要猜测"

    3. 常见 Prompt 技巧¶

    Zero-Shot：直接提问，不给示例 

    请把以下中文翻译成英文：你好

    Few-Shot：给 1-3 个示例 

    请把中文翻译成英文：
中文：苹果 -> 英文：apple
中文：香蕉 -> 英文：banana
中文：西瓜 ->

    Chain-of-Thought (CoT)：让 LLM 展示推理过程 

    问题：一个农场有鸡和兔，头共 35 个，脚共 94 只，鸡兔各多少？
请一步一步推理，先列出方程，再求解。

    ReAct Prompt：让 LLM 交替思考和行动 

    你是智能助手，解决问题时请按以下格式输出：
Thought: [你的思考]
Action: [你决定调用什么工具]
Observation: [工具返回的结果]

    4. AI 编程工具（面试高频！）¶

    工具 特点 面试关键词
    Cursor VSCode 插件，AI 原生 IDE Tab 补全、Cmd+K 编辑、Cmd+L 聊天
    Claude Code Anthropic 出品，命令行交互 Agent 模式、文件编辑、终端执行
    GitHub Copilot 微软出品，代码补全最强 单行补全、多行生成、聊天面板
    Windsurf Cascade 多文件编辑 跨文件重构、AI 规划执行

    面试常问： - 你用过哪些 AI 编程工具？对比优劣？ - 怎么确保 AI 生成代码的质量？（Code Review 流程） - AI 生成代码出错了怎么排查？

    5. AI 代码审查（CR）流程¶

    1. AI 生成初稿代码
2. 人工审查：逻辑正确性、边界情况、性能隐患
3. 运行测试：单元测试 + 集成测试
4. 发现问题 -> 回退给 AI 修复（带上下文）
5. 循环直到通过审查

    审查要点： - 变量命名是否合理 - 异常处理是否完善 - 是否有死循环/内存泄漏风险 - 是否符合团队编码规范

    二、面试真题 + 标准答案¶

    【真题 1】怎么设计一个好的系统提示词？（⭐⭐⭐⭐）¶

    答题模板： 1. 核心定义：系统提示词是告诉 LLM"你是谁、你能做什么、怎么做"的指令，决定了模型的行为基线。 2. 设计原则：角色明确（你是谁）+ 任务清晰（做什么）+ 输出规范（怎么输出）+ 边界约束（不能做什么）。 3. 具体做法： - 角色设定："你是一位资深后端工程师，擅长 Python 和系统设计" - 任务拆解：复杂任务分解为步骤，要求模型逐步输出 - Few-Shot：给 1-3 个输入输出示例，让模型模仿风格 - 输出格式：明确 JSON/XML/表格等格式要求，便于后续解析 4. 常见坑点：提示词太长导致上下文溢出；要求模糊导致输出不稳定；没有边界约束导致模型胡编乱造。 5. 生产优化：A/B 测试不同 Prompt 版本的效果；用评估框架（如 RAGAS）自动打分；建立 Prompt 版本管理（Git 追踪）。

    【真题 2】你用过哪些 AI 编程工具？对比优劣。（⭐⭐⭐⭐）¶

    答题模板： 1. 核心定义：AI 编程工具通过大模型辅助代码生成、补全、审查和重构，显著提升开发效率。 2. 工具对比： - Cursor：VSCode 深度集成，Agent 模式可以自主执行多文件修改，适合复杂重构场景；缺点是有时过度修改。 - Claude Code：命令行交互，对项目全局理解较好，适合大型代码库分析和批量修改；学习曲线较陡。 - Copilot：代码补全最流畅，单行/多行生成体验好；但复杂逻辑理解和跨文件能力较弱。 3. 工作流程：需求描述 -> AI 生成初稿 -> 人工 Code Review -> 测试验证 -> 迭代修复。 4. 常见坑点：AI 可能生成"看起来对但实际错"的代码；对业务逻辑理解不足；可能引入安全漏洞（如 SQL 注入）。 5. 生产优化：建立 AI 生成代码的审查清单；对核心模块强制人工 Review；保留 AI 修改前后的 diff 便于回滚。
