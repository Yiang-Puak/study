    02 | 记忆系统 —— 核心概念与面试答题模板¶

    一、核心概念速记¶

    1. 短期记忆 vs 长期记忆¶

    维度 短期记忆 长期记忆
    时间范围 当前会话内 跨会话持久化
    存储介质 上下文窗口 / list 向量数据库 / 文件 / Redis
    容量限制 受模型上下文长度限制（4K-128K） 理论上无上限
    实现方式 消息列表直接传入 向量存储 + 摘要 + 检索召回
    面试关键词 Sliding Window、Conversation Buffer VectorDB、RAG、Embedding

    2. 记忆管理策略（面试高频！）¶

    策略 原理 适用场景
    滑动窗口截断 只保留最近 N 轮对话 简单客服场景
    摘要压缩 每 K 轮生成摘要，替换原始对话 长会话场景
    重要信息提取 提取实体/意图存入结构化记忆 需要记住用户偏好的场景
    向量化存储 对话向量化存入 VectorDB，需要时检索 大规模跨会话记忆
    分层索引 热数据（内存）→ 温数据（Redis）→ 冷数据（磁盘/向量库） 高并发生产环境

    3. Token 限制下的记忆压缩策略（⭐⭐⭐⭐ 高频考点）¶

    问题：上下文越来越长，Token 超限怎么办？

方案 1：滑动窗口截断
- 只保留最近 N 轮（如 10 轮）
- 优点：简单、确定性强
- 缺点：丢失历史关键信息

方案 2：摘要压缩（Buffer + Summary）
- 早期对话 → 摘要
- 近期对话 → 原始消息
- 优点：保留核心信息，节省 Token
- 缺点：摘要可能丢失细节

方案 3：关键信息提取 + 向量化
- 从对话中提取关键实体/事件
- 向量化存入 VectorDB
- 需要时通过语义检索召回
- 优点：信息密度高，可跨会话
- 缺点：实现复杂，检索可能召回不准

方案 4：分层索引（Memory Hierarchy）
- 当前会话：内存中的消息列表
- 近期会话：Redis 缓存
- 历史会话：Chroma/Milvus 向量库
- 优点：性能与容量平衡
- 缺点：架构复杂，一致性难保证

    二、面试真题 + 标准答案（五步结构法）¶

    【真题 1】Agent 的短期记忆和长期记忆有什么区别？怎么实现？（⭐⭐⭐⭐⭐）¶

    答题模板： 1. 核心定义：短期记忆是会话级别的对话历史，长期记忆是跨会话持久化的知识/偏好存储。 2. 本质区别：短期记忆关注"这次对话说了什么"，长期记忆关注"用户长期偏好/历史事件是什么"。 3. 实现机制： - 短期记忆：用 Python list 维护 message 列表，传入 LLM 时受 max_tokens 限制，超限时用滑动窗口或摘要压缩。 - 长期记忆：关键信息提取后 Embedding 向量化，存入 Chroma/Milvus 等向量库；需要时做相似度检索召回；也可以存结构化数据（如用户画像表）。 4. 常见坑点：短期记忆不做压缩会导致 Token 暴涨、API 费用激增；长期记忆检索召回可能漏掉关键信息（需要调整 top_k 和阈值）。 5. 生产优化：短期记忆采用"Buffer + Summary"混合策略（近期原始 + 早期摘要）；长期记忆做定期清理/更新（TTL）；多用户场景下记忆必须隔离（user_id 做分区）。

    【真题 2】Token 限制下怎么压缩记忆？（⭐⭐⭐⭐）¶

    答题模板： 1. 核心定义：Token 压缩是指在 LLM 上下文长度受限的情况下，用更少 Token 保留更多有效信息。 2. 为什么必须做：Agent 多轮循环后历史消息线性增长，很快就会触及模型上下文上限（如 4K/8K/128K），导致无法输入或成本过高。 3. 压缩方案： - 滑动窗口：固定保留最近 N 轮，简单粗暴。 - 摘要替换：早期对话定期生成摘要，替换原始消息。LangChain 的 ConversationSummaryMemory 就是此方案。 - 实体提取：用 LLM 提取关键实体/事件/偏好，结构化存储，需要时注入上下文。 - 向量化检索：全部存入 VectorDB，每轮根据当前问题检索最相关的历史片段。 4. 常见坑点：摘要生成本身消耗 Token 和时间；滑动窗口会丢失早期关键信息；向量检索可能召回不相关内容。 5. 生产优化：混合策略——近期 5 轮保留原始消息，更早的生成摘要，用户画像等结构化信息单独存储；摘要生成用便宜的小模型；监控上下文 Token 使用量，触发压缩阈值可配置。

    【真题 3】大量记忆怎么处理？如何避免信息过载？（⭐⭐⭐）¶

    答题模板： 1. 核心定义：大量记忆处理是指在 Agent 运行过程中，历史信息规模膨胀时的存储、检索和筛选策略。 2. 问题本质：信息过载导致 LLM 注意力分散（大海捞针问题），同时 Token 成本和延迟暴涨。 3. 处理方案： - 分层存储：热数据（内存）→ 温数据（Redis，小时级 TTL）→ 冷数据（向量库，持久化）。 - 检索式召回：不把所有记忆都塞进上下文，而是根据当前 query 做语义检索，只召回最相关的 top_k 条。 - 重要性评分：给每条记忆打分，低分记忆定期归档或删除。 - 时间衰减：越久的记忆权重越低，除非被频繁访问。 4. 常见坑点：检索召回可能遗漏跨时间关联的信息；重要性评分规则难设计；时间衰减可能误删重要信息。 5. 生产优化：引入 Dendritic Network（或简化版关联图），让相关记忆自动关联；定期做记忆整理（类似人类"睡前回顾"）；用户主动标记"重要"的记忆永不过期。

    【真题 4】不同产品的记忆系统对比？（⭐⭐⭐ 加分项）¶

    答题模板： 1. 核心定义：不同 Agent 产品在记忆系统上的设计差异反映了它们的使用场景和技术取向。 2. 对比分析： - Hermès（类 ChatGPT）：对话式记忆，依赖大模型长上下文，近期推出"自我进化"机制——从用户反馈中自动提取偏好并更新记忆。 - OpenClaw 等框架：模块化记忆设计，支持多种后端（内存/Redis/VectorDB/数据库），灵活性高。 - LangChain Memory：提供 ConversationBuffer、Summary、Entity、Vector 等多种实现，开发者可按需组合。 3. 底层差异：Hermès 偏向"端到端黑盒"（模型自己决定记什么），LangChain 偏向"白盒可配置"（开发者显式定义记忆策略）。 4. 常见坑点：黑盒方案不可控，可能记住错误信息；白盒方案开发成本高。 5. 生产优化：推荐白盒为主、黑盒为辅——核心偏好用结构化存储，对话风格等软性记忆让模型自主学习。
