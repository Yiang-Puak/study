# 工程追问题库

> 这些题用来检验你是否真的能把知识点落到系统里。回答时优先讲链路、指标、失败点和取舍。

## Agent 架构

### 1. 什么场景需要 Agent，而不是普通工作流？

普通工作流适合步骤稳定、规则明确的任务。Agent 适合目标明确但路径不固定、需要动态选择工具、根据中间结果调整下一步的任务。

追问抓手：

- 任务是否需要规划和分支。
- 工具选择是否依赖中间结果。
- 是否需要记忆、反思或失败恢复。
- Agent 增加的不确定性是否值得。

### 2. Agent 跑偏怎么排查？

先看 trace，不要直接改 prompt：

1. 用户目标是否被正确理解。
2. 上下文是否混入无关内容。
3. 工具 schema 是否清楚。
4. 状态机是否缺少终止条件。
5. 错误是否被回填导致反复重试。

## RAG

### 3. RAG 回答不准怎么定位？

拆成五层：

1. Query：用户问题是否需要改写。
2. Chunk：切分是否丢语义或太长。
3. Retrieval：召回是否命中证据。
4. Rerank：排序是否把关键证据排前。
5. Generation：模型是否忠实引用证据。

指标可以用 Recall@k、MRR、Answer Relevancy、Faithfulness。

### 4. 什么时候用 Hybrid Search？

当关键词、专有名词、编号、字段名很重要时，纯向量召回可能不稳定。Hybrid Search 用 BM25 保留精确匹配，用向量召回语义相近内容，再用 rerank 统一排序。

## Tool / MCP

### 5. Tool Calling 的安全边界在哪里？

模型只负责提出“想调用什么工具和参数”。真正执行必须在应用侧完成：

- schema 校验
- 权限判断
- allowlist
- 超时和重试
- 高风险动作人工确认
- 审计日志

### 6. MCP 的价值是什么？

MCP 把工具、资源和上下文暴露方式协议化，让不同模型应用能用一致方式连接文件、数据库、浏览器、业务系统等外部能力。它解决的是工具接入碎片化问题，不是让模型自动变可靠。

## LangGraph / Workflow

### 7. LangGraph 相比普通链式调用强在哪里？

它显式建模 State、Node、Edge 和条件路由，适合多步、可回退、可观察的 Agent 工作流。面试时要强调它让状态流转、分支、重试和人工确认更清楚。

### 8. 什么时候不要用 LangGraph？

如果任务只是单轮问答、简单 RAG 或固定 API 调用链，用普通函数或 LCEL 更简单。LangGraph 的价值在复杂状态流转，不是所有场景都需要。

## Eval / Trace / Safety

### 9. 怎么证明 Agent 改动有效？

不要只展示一次 demo。要有：

- 固定 eval case
- pass rate
- 召回指标
- 工具调用成功率
- 延迟和 token 成本
- 失败分类

### 10. 线上出错怎么回放？

需要记录 request_id、conversation_id、node 输入输出、工具参数、模型响应、耗时、错误堆栈和最终结果。敏感数据要脱敏。

## Python 工程

### 11. Agent 项目为什么常用异步？

因为检索、LLM、数据库和工具调用大多是 I/O bound。异步不能让模型推理本身变快，但能减少多个 I/O 调用之间的等待，提高吞吐。

### 12. 怎么测试 LLM 应用？

分三层：

1. mock LLM 做单元测试。
2. 固定 case 做回归测试。
3. 真实模型做端到端评测并记录 trace。
