# Git 面试冲刺 —— 核心概念与面试答题模板

> 目标：从零理解 Git，能回答所有面试高频问题
> 适合：完全没用过 Git 的新手 + 需要面试复习的求职者

---

## 一、Git 是什么？（一句话理解）

**Git 就是一个"时间机器"，可以：**
- 记录你代码的每一次修改（拍照存档）
- 随时回到过去的任何一个版本（时光倒流）
- 多人协作不冲突（大家各自改，最后合并）

**类比：玩游戏时的存档系统**
- 你打到一个关卡，存个档（`git commit`）
- 打死了想重来，读档（`git checkout`）
- 朋友来你家，读你的存档继续打（`git clone`）
- 两个人同时打，最后合并进度（`git merge`）

## 二、Git 核心概念（面试必会）

### 1. 工作区、暂存区、版本库（⭐⭐⭐⭐⭐）

这是 Git 最基础也是面试最高频的概念！

```
┌─────────────────────────────────────┐
│  工作区（Working Directory）          │  ← 你看到的文件夹里的文件
│  你在 IDE 里编辑的代码就在这里         │
└──────────────┬──────────────────────┘
               │ git add
               ▼
┌─────────────────────────────────────┐
│  暂存区（Staging Area / Index）       │  ← 准备提交的快照
│  就像"准备上传的购物车"               │
└──────────────┬──────────────────────┘
               │ git commit
               ▼
┌─────────────────────────────────────┐
│  版本库（Repository）                  │  ← 永久保存的历史记录
│  .git 文件夹里，包含所有版本信息        │
└─────────────────────────────────────┘
```

**通俗理解**：
- **工作区** = 你书桌上的草稿纸（随意涂改）
- **暂存区** = 准备交给老师的作业（已经整理好了）
- **版本库** = 老师收走的作业档案（永久保存，有编号）

**面试常问**：为什么要有暂存区？
- 可以挑选性提交：只把改好的文件放购物车，没改好的不提交
- 可以分阶段准备：先 `git add` 一部分，再 `git add` 另一部分，最后一起 `git commit`

---

### 2. Git 对象模型（⭐⭐⭐⭐）

Git 本质上是一个"内容寻址文件系统"，里面有四种类型的对象：

| 对象类型 | 说明 | 类比 |
|----------|------|------|
| **Blob** | 文件内容（纯文本） | 一张试卷的内容 |
| **Tree** | 目录结构（文件名 + 对应 Blob） | 试卷袋上的目录清单 |
| **Commit** | 提交信息（作者、时间、父 Commit） | 存档时的备注条 |
| **Tag** | 给 Commit 起的别名（如 v1.0） | 给存档贴的标签 |

**面试金句**：Git 存储的是"快照"而不是"差异"。每次 commit，Git 都会保存当前所有文件的完整内容（通过引用已有 Blob 避免重复存储）。

---

### 3. 分支（Branch）（⭐⭐⭐⭐⭐）

**什么是分支？**
分支就是一条"独立的开发线"。你可以从主线上分出来，独自开发，最后再合并回去。

**类比：平行宇宙**
- 主线宇宙（`main`）：正常发展的世界
- 分支宇宙（`feature/login`）：你在这里实验新功能
- 实验成功了，把分支宇宙的内容合并到主线
- 实验失败了，直接删掉分支宇宙，主线不受影响

**分支的本质**：
分支只是一个"指向某个 Commit 的指针"。创建分支的成本极低（就创建一个指针），不像 SVN 那样复制整个代码库。

---

### 4. 合并（Merge）与变基（Rebase）（⭐⭐⭐⭐⭐）

**Merge（合并）**：
```
main:    A --- B --- C
              \
feature:       D --- E

merge 后:  A --- B --- C --- F(合并提交)
              \         /
               D --- E
```
- 保留完整历史，可以看到分支是怎么合进来的
- 会产生一个"合并提交"（Merge Commit）

**Rebase（变基）**：
```
main:    A --- B --- C
              \
feature:       D --- E

rebase 后:  A --- B --- C --- D' --- E'
```
- 把 feature 分支的提交"搬家"到 main 最新提交之后
- 历史变成一条直线，更干净
- **缺点**：会修改历史（D 变成 D'），如果已经 push 到远程，会造成混乱

**面试怎么选？**
| 场景 | 推荐 |
|------|------|
| 本地分支，还没 push | `rebase`（历史干净） |
| 已经 push 到公共仓库 | `merge`（不修改公共历史） |
| 不确定 | 默认用 `merge`，安全不出错 |

---

### 5. HEAD、索引、引用（⭐⭐⭐⭐）

- **HEAD**：一个特殊指针，指向"你当前所在的 Commit"
  - 通常指向当前分支的最新 Commit
  - `HEAD~1` 表示上一个 Commit，`HEAD~2` 表示上上个

- **索引（Index）**：就是"暂存区"的另一个名字

- **引用（Ref）**：所有以 `refs/` 开头的指针
  - `refs/heads/main`：本地 main 分支
  - `refs/remotes/origin/main`：远程 main 分支
  - `refs/tags/v1.0`：标签

## 三、高频命令速查表

### 基础操作（每天必用）

| 命令 | 作用 | 面试考点 |
|------|------|----------|
| `git init` | 初始化仓库 | 创建 `.git` 目录 |
| `git add 文件名` | 文件加入暂存区 | `git add .` 添加所有变更 |
| `git commit -m "说明"` | 提交到版本库 | `-m` 写清楚改了什么 |
| `git status` | 查看当前状态 | 红色 = 工作区有改动，绿色 = 已 staged |
| `git log` | 查看提交历史 | `--oneline` 简洁版，`--graph` 图形化 |
| `git diff` | 查看改动内容 | `git diff`（工作区 vs 暂存区），`git diff --cached`（暂存区 vs 版本库）|

### 分支操作

| 命令 | 作用 |
|------|------|
| `git branch` | 列出所有分支 |
| `git branch 新分支名` | 创建新分支 |
| `git checkout 分支名` | 切换到分支 |
| `git checkout -b 新分支名` | 创建并切换（常用！）|
| `git merge 分支名` | 把指定分支合并到当前分支 |
| `git branch -d 分支名` | 删除分支 |

### 远程操作

| 命令 | 作用 |
|------|------|
| `git clone 网址` | 克隆远程仓库 |
| `git remote -v` | 查看远程仓库地址 |
| `git pull` | 拉取远程更新（= fetch + merge）|
| `git push` | 推送本地提交到远程 |
| `git fetch` | 只下载远程更新，不合并 |

### 撤销与回退

| 命令 | 作用 | 危险程度 |
|------|------|----------|
| `git checkout -- 文件名` | 撤销工作区修改 | 低（未 add 的改动丢失）|
| `git reset HEAD 文件名` | 从暂存区撤出 | 低 |
| `git reset --soft HEAD~1` | 撤销最后一次 commit，改动保留在暂存区 | 中 |
| `git reset --mixed HEAD~1` | 撤销最后一次 commit，改动回到工作区 | 中 |
| `git reset --hard HEAD~1` | 彻底回退到上一个版本，改动全部丢失 | **高！慎用！** |

## 四、面试真题 + 标准答案

### 【真题 1】Git 和 SVN 有什么区别？（⭐⭐⭐⭐）

**答题模板**：
1. **核心区别**：Git 是分布式版本控制系统，SVN 是集中式。
2. **Git 分布式**：每个开发者本地都有完整的版本库（含所有历史），可以离线提交。远程仓库只是大家交换补丁的地方。
3. **SVN 集中式**：所有历史只存在中央服务器，开发者本地只有当前版本，提交必须联网。
4. **其他差异**：Git 分支操作极快（只是指针），SVN 分支慢（复制文件）；Git 存储快照，SVN 存储差异；Git 更适合开源协作，SVN 适合权限严格的企业内网。
5. **常见坑点**：初学者误以为 Git 必须有远程仓库才能工作，实际上本地可以完整使用。

---

### 【真题 2】git pull 和 git fetch 有什么区别？（⭐⭐⭐⭐）

**答题模板**：
1. **核心区别**：`fetch` 只下载不合并，`pull` = `fetch` + `merge`。
2. **`git fetch origin`**：把远程仓库的最新提交下载到本地的 `origin/main`，但你的 `main` 分支不动。你可以先看看有什么变化，再决定要不要合并。
3. **`git pull origin main`**：先 fetch 远程 main 的最新提交，然后自动 merge 到你当前的本地分支。
4. **什么时候用 fetch？** 想先看看远程改了什么，但还没准备好合并；或者担心直接 pull 会有冲突，想先评估。
5. **常见坑点**：pull 默认是 merge 策略，可能产生额外的 merge commit。可以用 `git pull --rebase` 让历史更干净。

---

### 【真题 3】merge 和 rebase 怎么选？（⭐⭐⭐⭐⭐）

**答题模板**：
1. **merge**：把两个分支的历史保留下来，产生一个合并提交。优点是历史完整，不会丢失任何提交；缺点是历史图比较乱。
2. **rebase**：把当前分支的提交"挪"到目标分支的最新提交之后，重写历史。优点是历史变成一条直线，非常干净；缺点是改变了原有提交的哈希值，如果已经 push 到公共仓库，会造成团队混乱。
3. **选择原则**：本地开发、还没 push 的分支用 rebase；已经 push 到公共仓库的分支用 merge。
4. **黄金法则**：不要对已经 push 到公共仓库的提交做 rebase。
5. **常见坑点**：rebase 过程中解决冲突更麻烦（可能要解决多次），merge 只解决一次。

---

### 【真题 4】git reset 的三种模式有什么区别？（⭐⭐⭐⭐）

**答题模板**：
1. **`--soft`**：只移动 HEAD 指针，暂存区和工作区都不变。撤销 commit 但保留改动在 staged 状态。适合：commit 写错了信息，想重新 commit。
2. **`--mixed`**（默认）：移动 HEAD 指针，清空暂存区，但工作区不变。撤销 commit，改动回到未 staged 状态。适合：想把已 commit 的改动拆分成多个小 commit。
3. **`--hard`**：移动 HEAD 指针，清空暂存区，强制把工作区也回退到指定版本。**改动会永久丢失！** 适合：彻底放弃当前所有改动（用之前务必确认）。
4. **安全建议**：`--hard` 是 Git 中最危险的命令之一，执行前可以用 `git stash` 备份当前改动。

---

### 【真题 5】什么是 Git 的" detached HEAD "状态？怎么恢复？（⭐⭐⭐）

**答题模板**：
1. **detached HEAD**：HEAD 直接指向某个 Commit，而不是指向分支名。通常发生在 `git checkout 某个commitID` 之后。
2. **在这个状态下**：你可以查看旧版本代码、做实验性修改，但如果直接 commit，这些提交不属于任何分支，切换分支后可能"丢失"。
3. **恢复方法**：
   - 如果想保留修改：`git checkout -b 新分支名`（基于当前 detached 状态创建新分支）
   - 如果不想保留：`git checkout 原分支名`（直接切回去）
4. **面试考点**：detached HEAD 不是错误，是 Git 的正常功能，但要注意 commit 后的处理。

---

### 【真题 6】如何解决 Git 冲突？（⭐⭐⭐⭐⭐）

**答题模板**：
1. **冲突产生原因**：两个人同时修改了同一个文件的同一行，Git 无法自动判断用哪个版本。
2. **解决步骤**：
   - `git status` 查看哪些文件有冲突
   - 打开冲突文件，找到 `<<<<<<< HEAD`、 `=======`、 `>>>>>>>` 标记
   - 手动编辑，保留正确的代码，删除冲突标记
   - `git add 文件名` 标记冲突已解决
   - `git commit` 完成合并
3. **冲突标记含义**：
   ```
   <<<<<<< HEAD
   当前分支的版本
   =======
   合并进来的版本
   >>>>>>> feature-branch
   ```
4. **减少冲突的技巧**：频繁 pull/push；小粒度 commit；避免多人同时改同一文件。

## 六、Git 面试实战练习题

### 练习 1：初始化仓库并提交

```bash
# 1. 新建文件夹并进入
mkdir my-project && cd my-project

# 2. 初始化 Git 仓库
git init

# 3. 创建文件
echo "Hello Git" > README.md

# 4. 查看状态（应该看到红色未跟踪）
git status

# 5. 加入暂存区
git add README.md

# 6. 查看状态（应该看到绿色已 staged）
git status

# 7. 提交
git commit -m "初始化项目，添加 README"

# 8. 查看历史
git log --oneline
```

### 练习 2：分支操作

```bash
# 1. 创建并切换到新分支
git checkout -b feature-login

# 2. 修改文件，然后 add + commit
echo "login功能" >> README.md
git add .
git commit -m "添加登录功能"

# 3. 切换回 main
git checkout main

# 4. 合并 feature-login
git merge feature-login

# 5. 删除已合并的分支
git branch -d feature-login
```

### 练习 3：撤销操作

```bash
# 场景 1：改错了文件，还没 add
git checkout -- 文件名

# 场景 2：已经 add 了，但还没 commit
git reset HEAD 文件名
git checkout -- 文件名

# 场景 3：已经 commit 了，但想修改 commit 信息
git commit --amend -m "新的提交信息"

# 场景 4：已经 commit 了，想回退到上一个版本（保留改动）
git reset --soft HEAD~1
```
