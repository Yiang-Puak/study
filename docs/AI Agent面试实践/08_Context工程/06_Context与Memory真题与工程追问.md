# Context 与 Memory 真题与工程追问

## 一、公开题型整理

| 题型 | 主要考察 |
| :--- | :--- |
| Context 与 Memory 边界 | 是否能区分容器、存储和召回 |
| 长会话压缩 | 是否会在预算、保真和恢复之间取舍 |
| 记忆写入策略 | 是否会区分事实、偏好、事件和临时草稿 |
| 记忆召回错误 | 是否能处理噪声、过期和权限 |
| 长任务恢复 | 是否知道 summary、notes、state 的组合 |

## 二、示范真题

### 题 1：一个代码 Agent 跑了很久，上下文里全是工具结果，你怎么收口

**回答骨架：**

1. 先分辨哪些是当前仍需推理的观察，哪些只是已完成步骤的原始输出。
2. 保留最近关键轮次、当前任务目标、约束、失败日志和未完成清单。
3. 对早期过程做 summary，对关键结论写结构化 notes。
4. 大块工具输出存引用或文件位置，不把全文永久留在上下文。
5. 压缩后用小回归检查确认任务状态没有丢。

### 题 2：长期记忆召回了过期偏好，回答开始跑偏，怎么修

**回答骨架：**

1. 给记忆加时间、来源、置信度和可更新机制。
2. 区分强事实与软偏好，软偏好要允许覆盖。
3. 召回前做用户、租户和任务相关过滤。
4. 在回答和 Trace 里保留记忆来源，便于排查错误注入。

### 题 3：RAG 和 Memory 都能检索，为什么还要分开设计

**先答结论：**因为检索对象、写入时机和使用风险不同。

| 维度 | RAG | Memory |
| :--- | :--- | :--- |
| 对象 | 文档、知识、证据 | 历史事件、偏好、任务状态 |
| 写入 | ingestion 或知识更新 | 交互后沉淀 |
| 目标 | 有依据回答 | 保持连续性和个性化状态 |
| 风险 | 错证据 | 错记忆、过期记忆 |

## 三、工程追问清单

| 追问 | 回答提示 |
| :--- | :--- |
| 摘要把关键参数丢了怎么办 | 关键事实结构化存储，摘要保引用 |
| 哪些记忆不该写入 | 敏感、低置信、临时噪声 |
| 长任务重启后怎么恢复 | notes、task state、recent summary |
| Memory 怎么评测 | 写入正确率、召回相关性、过期误用率 |
| Context 压缩触发条件怎么定 | token、轮数、延迟、工具输出体量 |
