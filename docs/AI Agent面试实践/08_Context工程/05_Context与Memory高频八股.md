# Context 与 Memory 高频八股

## 题单速刷

| 题目 | 先答什么 |
| :--- | :--- |
| Context 与 Memory 有什么区别 | Context 是当前输入，Memory 是可取回历史 |
| 长上下文还需要 Context Engineering 吗 | 需要，容量变大不等于信号密度更高 |
| 短期记忆和长期记忆怎么分 | 会话历史与跨会话持久信息 |
| 长会话怎么压缩 | 滑动窗口、摘要、结构化笔记、检索 |
| Tool Result 为什么要清理 | 观察重要，冗长原始结果不一定重要 |
| Memory 为什么不能全塞回上下文 | 会引入噪声、成本和错误召回 |

## 1. Context 与 Memory 的区别

**一句话答案：**Context 是模型本轮推理实际看到的信息集合，Memory 是为未来任务持久保存并按需取回的信息。

**展开顺序：**

1. Context 有 token 预算。
2. Memory 可以在上下文外存储。
3. Memory 被召回后才会进入 Context。
4. RAG 更偏外部知识检索，Memory 更偏历史状态、偏好、事件和任务经验。

## 2. 长上下文为什么还要做上下文工程

**一句话答案：**因为长窗口解决的是容量上限，不自动解决噪声、顺序、成本、权限和关键信息被淹没的问题。

**工程补充：**

- 指令、证据、历史和工具结果要争夺同一份预算。
- 不是所有工具原始输出都值得长期保留。
- 长任务仍需要摘要、笔记和状态持久化。

## 3. 长会话怎么压缩

| 策略 | 适合什么 |
| :--- | :--- |
| Sliding Window | 近期对话最重要，流程简单 |
| Summary | 需要保留早期结论与未完成任务 |
| Structured Notes | 状态、决策和 TODO 要可恢复 |
| Retrieval | 历史很多，只取当前相关片段 |

**记忆口诀：**近事保细节，远事保结论，未来按需取。

## 4. 短期记忆与长期记忆怎么落地

**30 秒版：**

短期记忆通常就是当前会话里的 messages、最近工具观察和任务状态，超预算后用窗口或摘要收缩。长期记忆应该写到上下文外，可以是结构化事实、事件摘要、向量化片段或任务笔记，后续通过检索和策略决定是否召回。

## 5. 高频误区

| 误区 | 修正 |
| :--- | :--- |
| Memory 越多越好 | 记忆要可选择、可更新、可遗忘 |
| Summary 永远可靠 | 摘要也会丢细节，要保留关键原文或引用 |
| RAG 和 Memory 一回事 | RAG 面向知识证据，Memory 面向历史与状态 |
| Tool Result 都该保留 | 保留观察结论和证据引用更稳 |
