# 01 | Tool 设计原则与容错

## 一句话定义

Tool 是 Agent 调用外部世界的执行接口。一个可靠 Tool 不只是函数本身，还包括清晰描述、严格参数、稳定输出、权限控制和失败兜底。

---

## Tool 的五要素

| 要素 | 说明 |
|------|------|
| Name | 工具名，必须简洁、语义明确 |
| Description | 工具用途，告诉 LLM 什么时候该调用 |
| Input Schema | 参数类型、必填项、枚举值、范围限制 |
| Execution | 实际执行逻辑，如 API、数据库、文件操作 |
| Output Contract | 返回格式，最好结构化，便于 LLM 继续推理 |

---

## 一个好 Tool Schema 的标准

```python
{
    "name": "search_documents",
    "description": "在内部知识库中检索与用户问题相关的文档片段。适合回答需要引用资料的问题，不适合闲聊。",
    "parameters": {
        "type": "object",
        "properties": {
            "query": {
                "type": "string",
                "description": "用户问题或检索关键词"
            },
            "top_k": {
                "type": "integer",
                "description": "返回文档数量，范围 1-10",
                "minimum": 1,
                "maximum": 10,
                "default": 5
            }
        },
        "required": ["query"]
    }
}
```

---

## 工具描述为什么重要

LLM 是否正确调用工具，很大程度取决于工具描述。描述要回答三件事：

1. **这个工具能做什么**
2. **什么时候应该调用它**
3. **什么时候不应该调用它**

错误示例：

```text
search: 搜索信息
```

更好示例：

```text
search_documents: 在内部知识库中检索资料。适合需要引用公司文档、技术规范、历史记录的问题；不适合实时互联网搜索。
```

## 工具调用容错机制

| 问题 | 方案 |
|------|------|
| 参数缺失 | JSON Schema 校验，缺失时要求模型补参或询问用户 |
| 参数非法 | 类型检查、范围检查、枚举检查 |
| 工具超时 | timeout + retry + fallback |
| 第三方 API 限流 | 指数退避、缓存、熔断 |
| 重复执行 | 幂等键 idempotency key |
| 危险操作 | Human-in-the-loop 二次确认 |
| 结果过长 | 结构化提取、摘要、截断 |

---

## 工具幂等性

幂等性指同一个请求执行一次和执行多次的效果一致。Agent 工具尤其需要幂等，因为模型可能因为失败重试而重复调用。

| 工具类型 | 是否需要幂等 | 示例 |
|----------|--------------|------|
| 查询类 | 一般天然幂等 | 查天气、查文档 |
| 写入类 | 强烈需要 | 创建工单、写数据库 |
| 支付类 | 必须保证 | 扣款、下单 |
| 文件修改类 | 需要版本检查 | 编辑文件、删除文件 |

## 面试答题模板

**问题：生产环境中工具调用失败怎么处理？**

1. **核心定义**：工具调用容错是让 Agent 在外部 API 失败、超时或返回异常时，不直接崩溃，而是可恢复、可降级。
2. **失败类型**：参数错误、网络超时、服务限流、权限不足、返回格式异常。
3. **处理方案**：参数校验、超时控制、指数退避重试、熔断、缓存、降级回答。
4. **关键风险**：重试可能造成重复写入，所以写操作要设计幂等键；危险操作要人工确认。
5. **生产优化**：记录工具调用 trace，监控成功率、延迟、错误类型，持续优化工具描述和参数 schema。
