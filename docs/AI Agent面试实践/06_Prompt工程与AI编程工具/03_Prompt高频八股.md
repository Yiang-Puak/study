# Prompt 工程高频八股

## 1. Prompt 工程和简单写提示词有什么区别？

简单写提示词关注“让模型这次回答好”。Prompt 工程关注可复用、可评测、可回归和可治理。

面试回答：

> 我会把 prompt 当成接口设计：定义输入上下文、输出 schema、边界约束和失败处理，再用固定 case 做回归测试。这样 prompt 不只是文案，而是 LLM 应用的一部分。

## 2. 怎样减少结构化输出失败？

1. 使用 JSON mode、function calling 或 schema parser。
2. Prompt 中给明确字段和示例。
3. 输出后做 parser 校验。
4. 失败时带错误信息重试。
5. 对关键字段做白名单校验。

## 3. Few-shot 示例越多越好吗？

不是。Few-shot 能提升模式稳定性，但会占上下文，也可能引入偏置。工程上要保留最能覆盖边界的少量样例，并用 eval case 验证收益。

## 4. Prompt 注入怎么防？

不能只靠一句“不要听用户恶意指令”。要分层处理：

- 用户输入和系统指令隔离。
- 工具调用参数做 schema 校验。
- 检索内容不直接获得系统权限。
- 高风险工具需要 allowlist 或人工确认。
- 记录 trace，方便审计。

## 5. Prompt 如何版本管理？

把 prompt 独立成模板文件或配置项，记录版本、适用场景、eval 指标和变更原因。上线前用固定 case 跑回归，不只看人工感觉。
