# Prompt 工程真题与工程追问

## 题 1：你怎么设计一个 Text-to-SQL Prompt？

回答结构：

1. 输入：用户问题、Schema RAG 结果、历史上下文、错误信息。
2. 输出：固定 JSON，包含 intent 和 sql。
3. 约束：只允许 SELECT、字段必须来自 schema、自动 LIMIT。
4. 失败恢复：SQL 执行报错后把错误回填给模型重试。
5. 评测：字段命中率、表召回率、执行成功率、延迟。

追问：如果模型生成了危险 SQL 怎么办？

> Prompt 只是第一层，真正安全要在执行器做拦截：只读连接、危险关键字过滤、多语句禁止、LIMIT、权限隔离。

## 题 2：Prompt 改完效果变差怎么排查？

1. 看是格式失败、事实失败、召回失败还是工具失败。
2. 对比改动前后的 eval case。
3. 查看 trace，确认输入上下文是否变化。
4. 回滚或拆分 prompt，减少一次性任务。

## 题 3：什么时候不用 prompt 解决问题？

当问题属于确定性逻辑、权限控制、数据校验或计算时，不应该交给 prompt。

例子：

- SQL 安全检查应该在 executor。
- 金额计算应该在代码或数据库。
- 权限判断应该在后端。
- 图表类型可以由规则或工具模块决定。

## 题 4：如何解释“Prompt 是接口”？

接口有输入、输出、契约和错误处理。Prompt 也一样：输入上下文、输出 schema、约束条件、parser、重试和 eval。这样才能工程化维护。
