# 01 | Agent 基础架构 —— 核心概念与面试答题模板

## 一、核心概念速记（面试前必背）

### 1. AI Agent 定义
**一句话**：Agent 是具备 **感知→规划→行动→记忆** 全闭环能力的自主智能体。

**与相似概念的区别**：
| 对比项 | LLM Chatbot | RAG 系统 | AI Agent |
|--------|-------------|----------|----------|
| 能力 | 仅对话生成 | 检索+生成 | 自主决策+调用工具+闭环执行 |
| 记忆 | 无/短会话 | 知识库检索 | 短期+长期记忆分层 |
| 行动 | 无 | 无 | 主动调用外部工具/API |
| 本质 | 文本生成器 | 信息增强器 | 任务执行器 |

### 2. Agentic Loop（Think → Act → Observe）

```
┌─────────────┐
│   用户输入   │
└──────┬──────┘
       ▼
┌─────────────┐
│  Think 思考 │ ← LLM 分析需求、拆解任务
└──────┬──────┘
       ▼
┌─────────────┐
│   Act 行动   │ ← 调用工具/API/执行代码
└──────┬──────┘
       ▼
┌─────────────┐
│ Observe 观察 │ ← 获取工具返回结果
└──────┬──────┘
       ▼
   是否完成？ ──否──→ 回到 Think
       │是
       ▼
   输出最终答案
```

### 3. ReAct 框架（Reasoning + Acting）
- **核心机制**：thought（推理）和 action（行动）交替进行
- **每轮只能做一件事**：先思考 → 再行动 → 再观察 → 再思考
- **解决什么问题**：让 LLM 在回答前能"停下来想想"，避免胡说八道

### 4. Function Call（工具调用协议）
- LLM 输出 JSON Schema 格式的函数调用请求
- 包含：函数名、参数（严格按定义的 schema）
- 系统执行后，将结果返回给 LLM 继续推理

### 5. Multi-Agent 协作
- **角色分工**：每个 Agent 有独立的 system prompt + 工具集
- **通信机制**：消息传递、共享状态池、主从协调
- **LangGraph 实现**：用图结构定义 Agent 流转，支持条件分支

### 6. MCP（Model Context Protocol）
- **是什么**：Anthropic 提出的标准协议，解决 LLM 与外部工具/数据源的标准化连接
- **解决什么问题**：之前每个 Agent 框架工具定义格式不统一，MCP 提供统一接口
- **面试热点**：字节、阿里 2026 年已考

## 二、面试真题 + 标准答案（五步结构法）

### 【真题 1】Agentic Loop 是什么？画出流程（⭐⭐⭐⭐⭐）

**答题模板**：
1. **核心定义**：Agentic Loop 是 Agent 的"思考→行动→观察"循环决策机制，是 Agent 自主解决任务的底层引擎。
2. **与相似概念区分**：不同于普通 LLM 的单轮生成，Agentic Loop 是多轮迭代，允许 LLM 在生成过程中调用外部工具获取信息。
3. **底层原理**：由 LLM 作为大脑驱动，每轮输出要么是最终答案（Finish），要么是工具调用请求（Action），系统执行后将 Observation 返回给 LLM，直到任务完成。
4. **常见坑点**：循环次数需设上限防止死循环；工具返回异常需容错处理；每轮只能调用一个工具，避免信息过载。
5. **生产优化**：设置 max_iterations 上限（如 10 次）；工具调用加超时控制；引入人类审核节点（Human-in-the-loop）。

---

### 【真题 2】AI Agent 和传统 LLM Chatbot 的本质区别是什么？（⭐⭐⭐⭐⭐）

**答题模板**：
1. **核心定义**：AI Agent 是具备自主感知、规划、行动、记忆闭环能力的任务执行智能体；Chatbot 是基于 LLM 的对话生成系统。
2. **本质差异**：Chatbot 的能力边界止于"生成文本"，Agent 的能力边界扩展到"完成任务"——它可以调用 API、操作数据库、写文件、发邮件。
3. **底层支撑**：Agent 依赖 ReAct/Function Call 机制连接外部工具，依赖记忆系统维持跨会话状态，依赖 Agentic Loop 实现多步推理。
4. **常见坑点**：不是加了工具的 Chatbot 就是 Agent，关键是"自主决策"能力——Agent 自己决定什么时候调用什么工具，而不是人类预设流程。
5. **生产优化**：Agent 需要完善的监控（调用链路追踪）、工具调用置信度阈值过滤、异常降级到纯 LLM 对话。

---

### 【真题 3】ReAct 框架的核心原理是什么？（⭐⭐⭐⭐）

**答题模板**：
1. **核心定义**：ReAct = Reasoning（推理）+ Acting（行动），是一种让 LLM 交替进行思考和行动的问题解决框架。
2. **与 Function Call 的关系**：ReAct 是"策略框架"，Function Call 是"执行协议"；ReAct 可以用 Function Call 作为行动手段。
3. **底层机制**：每轮 LLM 输出 Thought（我在想什么）→ Action（我要做什么）→ Observation（我看到了什么），循环直到产生 Final Answer。
4. **常见坑点**：LLM 可能产生格式错误的 Action，需要严格的输出解析；多工具场景下需限制每轮只调一个工具；推理链过长导致 token 消耗过大。
5. **生产优化**：引入 Reranker 对 Thought 质量评分；对推理链做摘要压缩；关键决策节点加入人工审核。

---

### 【真题 4】你了解 MCP 吗？它解决了什么问题？（⭐⭐⭐）

**答题模板**：
1. **核心定义**：MCP（Model Context Protocol）是 Anthropic 提出的模型上下文协议，标准化 LLM 与外部工具、数据源的连接方式。
2. **与现有方案的区别**：之前 LangChain Tools、OpenAI Functions、Google Functions 各家定义不统一，MCP 提供统一的 Server-Client 架构和工具描述规范。
3. **底层原理**：MCP Server 封装工具实现，通过标准协议暴露工具描述（名称、参数 schema、描述）；MCP Client（Agent）发现工具并调用。
4. **常见坑点**：MCP 目前生态还在建设中，不是所有工具都有 MCP Server；本地部署 MCP Server 需要额外维护。
5. **生产优化**：对 MCP Server 做健康检查；工具调用结果缓存；多 MCP Server 负载均衡。
