# Workflow、LangChain 与 LangGraph 学习页

> 先用本页拆概念，再回 [专题入口](index.md) 走完整闭环；要练短答案可直接去 [高频八股](03_Workflow与LangGraph高频八股.md)。

## 一句话定义

LangChain 更像大模型应用组件和组合接口，LangGraph 更适合把状态、分支、循环和 Agent 节点组织成图式 Workflow。

## 先回答选择题

| 你面对的问题 | 优先考虑 |
| :--- | :--- |
| 固定几步组件串联 | Chain |
| 任务步骤稳定但有分支 | Workflow |
| 需要根据观察动态选工具 | Agent Node |
| 有循环、恢复、人审 | LangGraph |
| 子任务天然独立并行 | Multi-Agent |

## 一、核心概念速记

### 1. LangChain 核心组件（⭐⭐⭐⭐）

| 组件 | 作用 | 面试关键词 |
|------|------|------------|
| **PromptTemplate** | 管理和复用提示词模板 | 参数化、多版本管理 |
| **ChatModel / LLM** | 封装各种模型调用（OpenAI/通义/本地） | 统一接口、可替换 |
| **OutputParser** | 解析模型输出（JSON/结构化/字符串） | Pydantic、容错解析 |
| **Retriever** | 检索相关文档 | VectorStore、BM25 |
| **Memory** | 管理对话历史 | Buffer、Summary、Entity |
| **Tools** | 外部工具封装 | @tool 装饰器、Schema 定义 |
| **Chain** | 组件串联的管道 | LCEL、Runnable |
| **Agent** | 自动决策调用工具的代理 | ReAct、Plan-and-Execute |

### 2. LangChain vs LangGraph（⭐⭐⭐⭐⭐ 必会对比）

| 维度 | LangChain | LangGraph |
|------|-----------|-----------|
| **定位** | 组件库 + 链式编排 | 复杂工作流 / 多 Agent 图编排 |
| **结构** | 线性 Chain（管道） | 图结构（节点 + 边 + 条件分支） |
| **状态管理** | 隐式传递 | 显式 State 管理 |
| **循环支持** | 不支持原生循环 | 支持循环、条件跳转 |
| **多 Agent** | 需手动协调 | 原生支持多 Agent 节点 |
| **适用场景** | 简单 RAG、单轮工具调用 | 复杂决策流、多 Agent 协作、审批流程 |
| **面试金句** | "LangChain 是乐高积木" | "LangGraph 是流程图引擎" |

### 3. LCEL（LangChain Expression Language）（⭐⭐）

- **本质**：用 `|` 操作符串联 Runnable 组件的声明式语法
- **示例**：`chain = prompt | model | parser`
- **优势**：可组合、支持流式、类型安全、可自动并行化
- **底层**：每个组件实现 `invoke()` / `stream()` / `batch()`，通过 `|` 自动衔接输入输出

### 4. 模块化 Chain 设计（⭐⭐⭐）

- **核心思想**：将检索、生成、后处理拆分为独立组件
- **依赖注入**：PromptTemplate 作为配置管理，运行时注入变量
- **好处**：组件可独立测试、替换、复用

## 二、面试真题 + 标准答案（五步结构法）

### 【真题 1】LangChain 和 LangGraph 有什么区别？什么时候用哪个？（⭐⭐⭐⭐⭐）

**答题模板**：
1. **核心定义**：LangChain 是大模型应用开发框架，提供组件化和链式编排；LangGraph 是 LangChain 生态的扩展，专门解决复杂工作流和多 Agent 协作的图编排问题。
2. **本质区别**：LangChain 适合"线性管道"（A → B → C），LangGraph 适合"复杂流程图"（分支、循环、多节点协作）。
3. **底层机制**：LangChain 用 LCEL 的 `|` 串联组件，数据隐式流转；LangGraph 显式定义 State（状态）、Nodes（节点/Agent）、Edges（流转边），支持条件分支和循环。
4. **选择原则**：
   - 用 LangChain：简单 RAG、单轮工具调用、固定流程的数据处理管道
   - 用 LangGraph：多 Agent 协作、需要循环迭代的任务、带条件分支的审批流、人机协作（Human-in-the-loop）
5. **生产优化**：LangChain Chain 可用 `astream_events` 做实时监控；LangGraph 用 `StateGraph` 的 `checkpoint` 做断点续传和状态回滚。

---

### 【真题 2】LCEL 是什么？有什么优势？（⭐⭐⭐）

**答题模板**：
1. **核心定义**：LCEL = LangChain Expression Language，是 LangChain 的链式声明语法，用 `|` 操作符串联 Runnable 组件。
2. **与传统写法对比**：传统写法是嵌套函数调用（难以阅读和调试），LCEL 是管道式声明（清晰、直观）。
3. **底层原理**：每个组件（PromptTemplate、ChatModel、Parser）都实现 `Runnable` 接口，包含 `invoke()`、`stream()`、`batch()` 方法；`|` 运算符本质是将上一个组件的输出映射为下一个组件的输入。
4. **优势**：可组合（组件像乐高一样拼接）、支持流式输出（`.stream()` 自动透传）、类型推断、自动并行化（`batch()` 时各组件自动并行）。
5. **生产优化**：复杂 Chain 建议拆分为子 Chain，用 `RunnableParallel` 并行执行独立分支；用 `astream_events` 输出中间节点日志，便于调试和监控。

---

### 【真题 3】怎么设计一个模块化的 RAG Chain？（⭐⭐⭐）

**答题模板**：
1. **核心定义**：模块化 RAG Chain 将索引、检索、增强、生成拆分为独立、可替换的组件。
2. **设计原则**：每个组件只负责一件事，通过标准接口（Runnable）衔接，支持依赖注入式配置。
3. **实现结构**：
   - `indexer.py`：文档加载 + Chunk + Embedding + 入库
   - `retriever.py`：查询改写 + 混合检索 + Reranker
   - `prompt_manager.py`：多版本 PromptTemplate 管理
   - `generator.py`：LLM 调用 + 输出解析
   - `pipeline.py`：用 LCEL 串联：`retriever | prompt | model | parser`
4. **常见坑点**：组件之间数据格式不一致（如检索器返回 Document 列表，Prompt 需要字符串）；缺乏统一的错误处理；Prompt 版本管理混乱。
5. **生产优化**：每个组件独立单元测试；用接口抽象（如 `BaseRetriever`）支持多种检索策略切换；Prompt 版本用 Git 管理；接入链路追踪（如 LangSmith）。

---

### 【真题 4】LangGraph 的状态管理是怎么实现的？（⭐⭐⭐）

**答题模板**：
1. **核心定义**：LangGraph 用显式的 `State` 对象管理工作流中的共享数据，State 是一个 TypedDict 或 Pydantic BaseModel。
2. **为什么需要**：多 Agent 协作时，每个 Agent 需要读取其他 Agent 的输出，隐式传递容易出错，显式 State 让数据流清晰可见。
3. **实现机制**：
   - 定义 State 结构：`class State(TypedDict): messages: list; documents: list; next_step: str`
   - 节点函数签名：`def agent_node(state: State) -> State:`，输入 State，修改后返回新 State
   - 边定义：普通边（固定流转）+ 条件边（`router` 函数根据 State 决定下一步去哪个节点）
4. **常见坑点**：State 是字典引用，原地修改可能导致不可预期的副作用；节点间传递大对象（如整份文档）导致内存和序列化开销；循环边需要设置退出条件否则死循环。
5. **生产优化**：State 用 Pydantic 做校验；大对象存引用（如文档 ID）而非内容；接入 `checkpointer` 实现断点续传和重试；用 `interrupt` 实现 Human-in-the-loop 审批。

## 三、常见误区

| 误区 | 修正 |
| :--- | :--- |
| 学会框架 API 就会编排 | 先明确状态、分支和失败路径 |
| LangGraph 就等于 Multi-Agent | 单 Agent Workflow 也很适合用图表达 |
| Agent 越自由越智能 | 任务边界越清楚越容易可靠 |
