# 高频追问题库

> 用法：先自己答，再展开看结构。每道题都要补一个项目例子。

---

??? question "1. 如果 RAG 检索错了怎么办？"
    分层排查：query 是否需要改写、chunk 是否合理、embedding 是否适配、TopK 是否太小、是否需要 BM25 混合检索、是否需要 Reranker。检索型系统还要拆开看召回指标和回答忠实度。

??? question "2. 为什么 SQL 能执行但答案仍然可能错？"
    因为 SQL 执行成功只代表语法和字段大概率正确，不代表业务口径正确。比如退款率问题没用 refunds 表也可能跑通，但答案错。

??? question "3. Agent 怎么避免无限循环？"
    设置最大迭代次数、最大重试次数、失败动作记录、错误处理节点和人工兜底。每个循环节点都要有退出条件。

??? question "4. 多 Agent 一定比单 Agent 好吗？"
    不一定。多 Agent 会增加通信、调度、状态同步和错误传播成本。应先把单 Agent 做强，只有当工具太多、角色明显、任务天然分工时再拆。

??? question "5. Tool Schema 怎么设计才好？"
    工具名清晰、描述具体、参数少而准、类型明确、返回结构化、错误可读。高风险工具要标注权限和审批要求。

??? question "6. Trace 和普通日志有什么区别？"
    普通日志通常是离散文本，Trace 关注一次 Agent run 的完整链路，包括模型调用、工具调用、状态变化、耗时和失败点。

??? question "7. 怎么做成本优化？"
    减少无效上下文、缓存检索结果、用小模型做分类和路由、强模型只做关键决策、批量离线评测、限制最大工具调用轮数。

??? question "8. 什么时候需要长期记忆？"
    当系统需要跨会话记住用户偏好、业务口径、常用指标或历史任务时。短期会话追问只需要 session memory，不必过早引入长期记忆。

??? question "9. Prompt Injection 怎么防？"
    不能只靠 system prompt。要做输入隔离、工具权限分级、只读优先、高危动作人审、输出校验和审计日志。

??? question "10. 你项目里最大的工程取舍是什么？"
    可以答：先用最小可评测链路验证问题，再逐步加复杂能力。比如先做单路检索和引用，再根据失败样本决定是否引入混合检索、reranker 或 query rewrite。

??? question "11. 为什么要自动生成评测报告？"
    Markdown 面向人，适合面试展示和复盘；JSON 面向机器，后续可接 CI、Dashboard 和版本对比。

??? question "12. 知识库规模扩到大量文档后怎么办？"
    做分层索引、元数据过滤、混合检索和 reranker；复杂问题可先路由数据域，再召回证据，并用离线评测集看召回质量是否下降。

??? question "13. 怎么证明你不是只会调 Prompt？"
    用项目证据：后端 API、SQL 安全执行器、LangGraph 状态机、前端可观测、Eval Harness 自动报告、构建和运行文档。

??? question "14. 为什么企业级 Agent 必须有权限和审计？"
    因为 Agent 可能调用真实业务系统，错误动作会造成数据、财务或合规风险。权限和审计能追踪谁在什么时间触发了什么动作。

??? question "15. 怎么把 Agent 项目讲得不像只是在调 Prompt？"
    先讲链路，再讲证据：后端 API、工具边界、状态机、RAG 检索、安全执行、Trace 和 Eval Harness。重点说明每个节点解决什么失败模式，以及用什么指标验证改动。
