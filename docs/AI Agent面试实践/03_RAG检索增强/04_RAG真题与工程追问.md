# RAG 真题与工程追问

> 真题页不追求背长答案，而是训练“先判断故障层，再给工程动作”。

## 一、公开题型整理

| 题型 | 面试官在看什么 | 推荐先答 |
| :--- | :--- | :--- |
| 画 RAG 链路 | 是否真的知道数据怎么流 | 离线索引与在线问答分开画 |
| Chunk 取舍 | 是否理解证据粒度 | 语义完整性与检索精度平衡 |
| 长上下文是否替代 RAG | 是否会区分上下文容量和知识治理 | 成本、时效、引用、权限、评测 |
| RAG 效果差怎么排查 | 是否有误差分析思路 | 检索、排序、上下文、生成分层定位 |
| 引用不准怎么办 | 是否关注 grounded answer | 证据绑定、片段 ID、引用校验 |

## 二、示范真题

### 题 1：一个知识库问答系统回答错了，你怎么判断是检索问题还是生成问题

**回答骨架：**

1. 先查 Trace，固定同一条 query 的召回候选、rerank 结果和最终上下文。
2. 如果正确证据根本没有进入候选集，优先看清洗、chunk、metadata、query 改写和召回策略。
3. 如果证据进入候选但排不到前面，优先看融合排序、reranker 和过滤规则。
4. 如果证据已经进入最终上下文仍答错，优先看 Prompt 约束、上下文噪声、引用绑定和拒答策略。
5. 最后把这条失败样本沉淀进回归集，避免只修一个 case。

### 题 2：长上下文模型越来越强，为什么还要 RAG

**先答结论：**长上下文扩大了可输入内容，不等于自动解决知识更新、权限过滤、引用定位和检索成本。

**工程追问方向：**

- 哪些资料适合整包放入上下文，哪些资料必须检索。
- 权限过滤应该发生在检索前、检索中还是生成后。
- 长上下文和 RAG 混用时怎样做证据压缩。

### 题 3：你会怎样做一个可复盘的 RAG 优化实验

**回答骨架：**

1. 固定评测集：问题、期望证据、答案要点、拒答样本。
2. 固定变量：一次只改 chunk、召回、rerank 或 Prompt 中的一类。
3. 对比检索指标与答案指标，不能只看最终主观满意度。
4. 保留 Trace 与版本号，让优化结果可复现。

## 三、工程追问清单

| 追问 | 回答提示 |
| :--- | :--- |
| 文档更新后旧答案缓存怎么办 | 建版本、失效缓存、增量索引 |
| OCR 或表格导致 chunk 乱了怎么办 | 先修 ingestion 与结构化抽取 |
| 为什么 top_k 不是越大越好 | 噪声、成本、上下文预算 |
| 什么时候先加 metadata filter | 权限、租户、时间、文档类型强约束 |
| 什么时候做 query rewrite | 问题歧义大、口语化强、召回不足 |
| 没有标注集怎么起步 | 从高价值问题建小评测集，再逐步补失败样本 |
