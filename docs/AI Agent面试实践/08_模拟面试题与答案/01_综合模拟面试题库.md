## 一、Agent 基础架构（⭐⭐⭐⭐⭐ 必考）

### 题目 1.1
**Q：Agentic Loop 是什么？画出流程。**

**模拟答题（你先口述，再展开看答案）**

<details>
<summary>点击查看参考答案</summary>

1. **核心定义**：Agentic Loop 是 Agent 的"思考→行动→观察"循环决策机制，让 LLM 能在任务执行过程中迭代推理、调用工具、获取反馈，直到任务完成。
2. **与相似概念区分**：不同于单轮 LLM 的一次性生成，Agentic Loop 是多轮交互；不同于固定流程的脚本，Agentic Loop 中每一步的 Action 由 LLM 自主决定。
3. **底层原理**：每轮 LLM 接收当前状态（用户请求 + 历史动作 + 工具返回），输出 Thought（推理过程）和 Action（工具调用或最终答案），系统执行 Action 后将 Observation 返回给 LLM，循环直到产生 Final Answer 或达到最大迭代次数。
4. **常见坑点**：循环次数不设上限会导致死循环和 Token 爆炸；LLM 可能产生格式错误的 Action，需要严格的解析和容错；一步推理错误会导致后续全错。
5. **生产优化**：设置 max_iterations（如 5-10）；工具调用加超时和重试；关键节点引入 Human-in-the-loop；对工具结果做置信度过滤。

**加分项**：能画出流程图（Think → Act → Observe 循环，带退出判断）。
</details>

### 题目 1.2
**Q：AI Agent 和传统 LLM Chatbot 的本质区别是什么？**

<details>
<summary>点击查看参考答案</summary>

1. **核心定义**：AI Agent 是具备自主感知、规划、行动、记忆闭环能力的任务执行智能体；Chatbot 是基于 LLM 的对话生成系统。
2. **本质差异**：Chatbot 的能力边界止于"生成文本"，Agent 的能力边界扩展到"完成任务"——它可以自主调用 API、操作数据库、写文件。
3. **底层支撑**：Agent 依赖 ReAct/Function Call 机制连接外部工具，依赖记忆系统维持跨会话状态，依赖 Agentic Loop 实现多步推理。
4. **常见坑点**：不是加了工具的 Chatbot 就是 Agent，关键是"自主决策"能力——Agent 自己决定什么时候调用什么工具，而不是人类预设固定流程。
5. **生产优化**：Agent 需要完善的监控（调用链路追踪）、工具调用置信度阈值过滤、异常降级到纯 LLM 对话。
</details>

### 题目 1.3
**Q：ReAct 框架的核心原理是什么？与 Function Call 有什么关系？**

<details>
<summary>点击查看参考答案</summary>

1. **核心定义**：ReAct = Reasoning（推理）+ Acting（行动），是一种让 LLM 交替进行思考和行动的问题解决框架。
2. **与 Function Call 的关系**：ReAct 是"策略框架"，Function Call 是"执行协议"；ReAct 可以用 Function Call 作为行动手段，但 Function Call 也可以独立于 ReAct 使用（如单次工具调用）。
3. **底层机制**：每轮 LLM 输出 Thought（我在想什么）→ Action（我要做什么）→ Observation（我看到了什么），循环直到产生 Final Answer。
4. **常见坑点**：LLM 可能产生格式错误的 Action，需要严格的输出解析；多工具场景下需限制每轮只调一个工具，避免信息过载；推理链过长导致 token 消耗过大。
5. **生产优化**：引入 Reranker 对 Thought 质量评分；对推理链做摘要压缩；关键决策节点加入人工审核。
</details>

### 题目 1.4
**Q：你了解 MCP 吗？它解决了什么问题？**

<details>
<summary>点击查看参考答案</summary>

1. **核心定义**：MCP（Model Context Protocol）是 Anthropic 提出的模型上下文协议，标准化 LLM 与外部工具、数据源的连接方式。
2. **与现有方案的区别**：之前 LangChain Tools、OpenAI Functions、Google Functions 各家定义不统一，MCP 提供统一的 Server-Client 架构和工具描述规范。
3. **底层原理**：MCP Server 封装工具实现，通过标准协议暴露工具描述（名称、参数 schema、描述）；MCP Client（Agent）发现工具并调用。
4. **常见坑点**：MCP 目前生态还在建设中，不是所有工具都有 MCP Server；本地部署 MCP Server 需要额外维护。
5. **生产优化**：对 MCP Server 做健康检查；工具调用结果缓存；多 MCP Server 负载均衡。
</details>

### 题目 1.5（进阶）
**Q：Multi-Agent 协作怎么设计？怎么避免循环依赖？**

<details>
<summary>点击查看参考答案</summary>

1. **核心定义**：Multi-Agent 是将复杂任务分解给多个专业 Agent 协作完成，每个 Agent 有独立的角色、工具集和记忆。
2. **设计要素**：角色分工（研究员/写手/审核员）、通信机制（消息传递/共享状态池/主从协调）、任务编排（Pipeline/图结构）。
3. **避免循环依赖**：通过中央协调器（Orchestrator）模式，Agent 之间不直接通信，所有消息都经过协调器转发，由协调器控制流程和状态同步。
4. **常见坑点**：状态一致性问题（多个 Agent 同时修改共享状态）；循环调用控制（A 调 B，B 调 A）；错误传播（一个 Agent 失败导致全链路失败）。
5. **生产优化**：LangGraph 的 StateGraph 显式管理状态和流转；每个 Agent 独立的错误处理和降级策略；引入超时和熔断机制。
</details>

## 二、记忆系统（⭐⭐⭐⭐）

### 题目 2.1
**Q：Agent 的短期记忆和长期记忆有什么区别？**

<details>
<summary>点击查看参考答案</summary>

1. **核心定义**：短期记忆是会话级别的对话历史，长期记忆是跨会话持久化的知识/偏好存储。
2. **本质区别**：短期记忆关注"这次对话说了什么"，长期记忆关注"用户长期偏好/历史事件是什么"。
3. **实现机制**：短期记忆用 Python list 维护 message 列表，传入 LLM 时受 max_tokens 限制，超限时用滑动窗口或摘要压缩；长期记忆关键信息提取后 Embedding 向量化，存入 Chroma/Milvus 等向量库。
4. **常见坑点**：短期记忆不做压缩会导致 Token 暴涨、API 费用激增；长期记忆检索召回可能漏掉关键信息。
5. **生产优化**：短期记忆采用"Buffer + Summary"混合策略；长期记忆做定期清理/更新（TTL）；多用户场景下记忆必须隔离（user_id 做分区）。
</details>

---

### 题目 2.2
**Q：Token 限制下怎么压缩记忆？**

<details>
<summary>点击查看参考答案</summary>

1. **核心定义**：Token 压缩是指在 LLM 上下文长度受限的情况下，用更少 Token 保留更多有效信息。
2. **为什么必须做**：Agent 多轮循环后历史消息线性增长，很快就会触及模型上下文上限。
3. **压缩方案**：滑动窗口（保留最近 N 轮）、摘要替换（早期对话生成摘要）、实体提取（结构化存储关键信息）、向量化检索（全部存入 VectorDB，按需召回）。
4. **常见坑点**：摘要生成本身消耗 Token 和时间；滑动窗口会丢失早期关键信息；向量检索可能召回不相关内容。
5. **生产优化**：混合策略——近期 5 轮保留原始消息，更早的生成摘要；摘要生成用便宜的小模型；监控上下文 Token 使用量。
</details>

---

### 题目 2.3（进阶）
**Q：大量记忆怎么处理？如何避免信息过载？**

<details>
<summary>点击查看参考答案</summary>

1. **核心定义**：大量记忆处理是指在 Agent 运行过程中，历史信息规模膨胀时的存储、检索和筛选策略。
2. **问题本质**：信息过载导致 LLM 注意力分散（大海捞针问题），同时 Token 成本和延迟暴涨。
3. **处理方案**：分层存储（热数据内存→温数据 Redis→冷数据向量库）、检索式召回（根据当前 query 做语义检索，只召回 top_k）、重要性评分、时间衰减。
4. **常见坑点**：检索召回可能遗漏跨时间关联的信息；重要性评分规则难设计；时间衰减可能误删重要信息。
5. **生产优化**：引入关联图让相关记忆自动关联；定期做记忆整理；用户主动标记"重要"的记忆永不过期。
</details>

## 三、RAG 检索增强（⭐⭐⭐⭐⭐ 必考）

### 题目 3.1
**Q：RAG 的完整流程是什么？画出链路图。**

<details>
<summary>点击查看参考答案</summary>

1. **核心定义**：RAG（Retrieval-Augmented Generation）= 检索 + 增强 + 生成，让 LLM 在回答前先查资料，减少幻觉。
2. **与纯 LLM 的区别**：纯 LLM 依赖训练参数中的知识（有截止日期、会幻觉）；RAG 让 LLM 动态访问外部知识库，答案更实时、可溯源。
3. **四阶段流程**：索引（文档→Chunk→Embedding→向量库）、检索（Query Embedding→相似度搜索→召回 top_k）、增强（将 chunk 拼入 Prompt）、生成（LLM 结合上下文生成答案）。
4. **常见坑点**：Chunk 切分不当导致语义割裂；检索召回不相关文档会污染生成结果；上下文过长超出模型限制。
5. **生产优化**：混合检索（语义+关键词）+ Reranker 精排 + 多级缓存 + 检索结果溯源。
</details>

---

### 题目 3.2
**Q：Embedding 怎么选？Chunk 怎么切？**

<details>
<summary>点击查看参考答案</summary>

1. **核心定义**：Embedding 是将文本映射到语义向量的过程，Chunk 是将长文档切分为适合检索和生成的片段。
2. **选型原则**：中文通用 BGE-M3/GTE-large；英文 E5/OpenAI；专业领域必须在领域语料上微调。
3. **Chunk 策略**：固定长度（简单但有语义割裂）、按段落/句子（保留自然边界）、重叠窗口（减少边界信息丢失）、语义切分（效果好但成本高）。
4. **常见坑点**：Chunk 太大→检索粒度粗；Chunk 太小→缺少上下文，LLM 生成质量差。
5. **生产优化**：多粒度索引（句子级+段落级）；Chunk 内加入标题/章节信息；对高频查询做 Embedding 结果缓存。
</details>

---

### 题目 3.3
**Q：什么是混合检索？α 怎么调？**

<details>
<summary>点击查看参考答案</summary>

1. **核心定义**：混合检索 = 语义检索（向量相似度）+ 关键词检索（BM25），两者结果融合后返回。
2. **为什么需要**：纯语义检索对专业术语匹配不准；纯关键词检索无法理解同义词。
3. **融合公式**：`score = α * vector_score + (1-α) * bm25_score`
4. **α 动态调整**：实体查询（查人名/法条）α 偏小（0.2-0.4）；分析查询（总结/解释）α 偏大（0.6-0.8）；可用小分类器判断 query 类型自动路由。
5. **生产优化**：路由分类后决定 α；融合后 Reranker 精排；记录不同 α 下用户满意度，A/B 测试调优。
</details>

---

### 题目 3.4（进阶）
**Q：Reranker 是什么？为什么需要两阶段检索？**

<details>
<summary>点击查看参考答案</summary>

1. **核心定义**：Reranker 是检索后的精排模型，对召回的大量候选结果重新打分排序。
2. **为什么需要两阶段**：第一阶段（召回）要求速度快，用轻量方法召回 top_100；第二阶段（精排）要求精准度高，用更强的交叉编码器精细打分，取 top_5/10。如果直接用精排做召回，计算量太大。
3. **典型模型**：bge-reranker、Cohere Rerank、Jina Reranker
4. **常见坑点**：Reranker 也有幻觉风险——给不相关文档打高分；精排模型输入长度有限，长文档需要截断。
5. **生产优化**：Reranker 结果加入置信度阈值；Reranker 结果也缓存；对高频 query 的精排结果做离线预计算。
</details>

## 四、LangChain / LangGraph（⭐⭐⭐⭐）

### 题目 4.1
**Q：LangChain 和 LangGraph 有什么区别？什么时候用哪个？**

<details>
<summary>点击查看参考答案</summary>

1. **核心定义**：LangChain 是大模型应用开发框架；LangGraph 是 LangChain 生态的扩展，专门解决复杂工作流和多 Agent 协作的图编排问题。
2. **本质区别**：LangChain 适合"线性管道"（A→B→C），LangGraph 适合"复杂流程图"（分支、循环、多节点协作）。
3. **底层机制**：LangChain 用 LCEL 的 `|` 串联组件，数据隐式流转；LangGraph 显式定义 State、Nodes、Edges，支持条件分支和循环。
4. **选择原则**：简单 RAG/单轮工具调用用 LangChain；多 Agent 协作/循环迭代/审批流用 LangGraph。
5. **生产优化**：LangChain 用 `astream_events` 做实时监控；LangGraph 用 `checkpoint` 做断点续传和状态回滚。
</details>

---

### 题目 4.2（进阶）
**Q：LangGraph 的状态管理是怎么实现的？**

<details>
<summary>点击查看参考答案</summary>

1. **核心定义**：LangGraph 用显式的 `State` 对象管理工作流中的共享数据，State 是一个 TypedDict 或 Pydantic BaseModel。
2. **为什么需要**：多 Agent 协作时，每个 Agent 需要读取其他 Agent 的输出，隐式传递容易出错，显式 State 让数据流清晰可见。
3. **实现机制**：定义 State 结构（包含 messages/documents/next_step 等字段）；节点函数签名 `def agent_node(state: State) -> State:`；普通边固定流转，条件边通过 router 函数根据 State 决定下一步。
4. **常见坑点**：State 是字典引用，原地修改可能导致副作用；节点间传递大对象导致内存开销；循环边需要设置退出条件否则死循环。
5. **生产优化**：State 用 Pydantic 做校验；大对象存引用而非内容；接入 checkpointer 实现断点续传；用 interrupt 实现 Human-in-the-loop。
</details>

## 五、综合场景题（⭐⭐⭐⭐⭐ 大厂爱考）

### 题目 5.1
**Q：设计一个电商客服 Agent，需要哪些模块？怎么实现？**

<details>
<summary>点击查看参考答案</summary>

**架构设计**：
1. **意图识别模块**：用 LLM 或分类器判断用户意图（查物流/退换货/商品咨询/投诉）
2. **检索模块**：RAG 检索商品 FAQ、订单信息、物流状态
3. **工具模块**：查订单 API、申请退款 API、转人工 API
4. **记忆模块**：记住用户历史订单、偏好、投诉记录
5. **生成模块**：结合检索结果和记忆生成回复

**实现方案**：
- LangChain 做 RAG 和工具调用
- LangGraph 编排复杂流程（如"投诉→审核→退款"需要多节点审批）
- 向量库存商品知识和历史对话
- Redis 缓存热点问题答案

**生产要点**：
- 敏感操作（退款/改地址）需要 Human-in-the-loop 确认
- 接入监控统计解决率、平均响应时间、用户满意度
- 无法解决的问题自动转人工，并携带完整上下文
</details>

### 题目 5.2（进阶）
**Q：你的 RAG 项目检索效果不理想，怎么排查和优化？**

<details>
<summary>点击查看参考答案</summary>

**排查思路（从近到远）**：
1. **Prompt 层**：检查上下文是否拼入正确？是否有"忽略参考资料"的提示泄漏？
2. **生成层**：LLM 是否理解了上下文？换更强的模型测试
3. **检索层**：召回的文档是否与 query 相关？Hit@K 多少？
4. **索引层**：Chunk 切分是否合理？Embedding 质量如何？
5. **数据层**：知识库是否包含答案？数据质量如何？

**优化方案**：
- 检索层：加 BM25 混合检索 + Reranker 精排 + 查询改写
- 索引层：优化 Chunk 策略、微调 Embedding、加文档标题/标签
- 数据层：补充缺失文档、清洗低质量数据
- 系统层：加多级缓存、异步索引更新、检索结果溯源
</details>
