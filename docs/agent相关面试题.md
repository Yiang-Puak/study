## 一、Agent 基础架构与核心概念

### Q1：什么是 AI Agent？它和传统 LLM Chatbot 有什么本质区别？

**有工作经验视角：**

Agent 和 Chatbot 的本质区别在于 **自主决策 + 行动能力**。

- Chatbot：用户输入 → LLM 推理 → 输出文本，是被动的问答机。
- Agent：用户输入 → LLM 推理 → 决定用什么工具 → 执行工具 → 观察结果 → 决定下一步 → 循环直到目标达成。

以客服 Agent 为例，核心是增加了一个“决策中枢”——模型在每一轮判断：这个话题我能否直接回答？是否需要查订单系统？查完后用户满意吗？不满意是否转人工？这套决策逻辑与一问一答完全不同。

Agent 必须具备四个核心能力：
- **感知（Perception）**：理解多模态输入（文本、图片、语音）
- **规划（Planning）**：把复杂任务拆解为可执行子任务
- **行动（Action）**：调用工具/API/外部系统
- **记忆（Memory）**：跨对话轮次的信息保持

生产环境的最大坑：自主性是把双刃剑。给太多自主权，模型可能做出不可预期的操作；给太少，则退化成 Chatbot。真正的挑战在于设计安全的自主边界。

**背面经视角：**

| 维度 | Chatbot | Agent |
|------|---------|-------|
| 能力 | 只生成文本 | 生成文本 + 调用工具 |
| 交互 | 一问一答 | 多步骤自主循环 |
| 记忆 | 通常无状态 | 短期 + 长期记忆 |
| 核心组件 | LLM | LLM + Planning + Memory + Tools |

Agent 的核心循环为 **Think → Act → Observe**，经典框架是 ReAct（Reasoning + Acting）。

### Q2：什么是 Agentic Loop？请画出基本流程。

**高频来源：字节跳动、阿里、腾讯**

**答案：**

Agentic Loop 是 Agent 执行任务的核心循环，通常包含三个阶段：
用户目标 → [思考(Think)] → [行动(Act)] → [观察(Observe)] → 判断是否完成
↑ ↓
└──────────── 未完成则继续循环 ──────────┘

1. **Think**：LLM 分析当前状态，生成下一步计划（可能包括调用哪个工具、传入什么参数）。
2. **Act**：执行选择的工具（API 调用、数据库查询、代码执行等）。
3. **Observe**：收集工具返回结果，更新上下文记忆。
4. **Judge**：判断任务是否完成，若未完成则回到 Think 阶段。

**实例**：用户要求“帮我整理上个月的销售报告并发送邮件”。Agent 会：
- Think：先查询数据库获取销售数据
- Act：执行 SQL 查询
- Observe：得到数据结果
- Think：数据拿到了，现在需要生成报告图表
- Act：调用绘图工具生成图表
- Observe：图表生成成功
- Think：现在用邮件发送
- Act：调用邮件 API 发送
- Observe：发送成功，任务完成

### Q3：描述 ReAct 框架的工作原理，它和 Function Call 有何不同？

**答案：**

ReAct（Reasoning + Acting）是 Google 2022 年提出的框架，核心思想是将**推理（Thought）**与**行动（Action）**交替进行。
Thought → Action → Observation → Thought → Action → ...

- **Thought**：模型生成自然语言推理，分析当前状态并决定下一步。
- **Action**：模型输出可执行的动作指令（如搜索、计算器）。
- **Observation**：环境返回动作结果。

**与 Function Call 的区别**：
- ReAct 是一种**提示工程策略**，通过 Few-shot 样例让模型学会交替输出推理和动作，本质是文本生成。
- Function Call 是**模型能力**，通过模型微调使其能输出结构化 JSON 来描述要调用的函数及参数，更稳定、更易解析。
- 实际应用中，Function Call 是 ReAct 的底层支撑技术，两者不矛盾。

### Q4：什么是 MCP（Model Context Protocol）？它解决了什么问题？

**答案：**

MCP 是 Anthropic 提出的开放协议，旨在**标准化 LLM 与外部工具、数据源之间的连接方式**。

**核心问题**：传统 Agent 开发中，每接入一个新工具（数据库、API、文件系统）都需要定制化开发，造成 M×N 集成难题。

**MCP 的解决方式**：
- 定义统一的 Client-Server 架构
- 所有工具都以 MCP Server 形式提供，Agent 作为 MCP Client 通过标准接口调用
- 支持 **Tools**（动作执行）、**Resources**（数据读取）、**Prompts**（提示模板）三种原语

**面试亮点**：字节、阿里等大厂 Agent 岗位已在询问 MCP，可以提及它是“AI 集成的事实标准”，类似于 USB 协议对硬件的统一。

### Q5：Skill 是什么？由哪几部分组成？

**答案：**

Skill 是 Agent 的**可复用功能单元**，封装了完成特定任务所需的知识、流程和接口。

**组成部分**：
1. **名称（Name）**：唯一标识，如 "send_email"
2. **描述（Description）**：自然语言说明，用于 Agent 决策选择哪个 Skill
3. **输入参数（Input Schema）**：JSON Schema 定义参数类型、必填项
4. **执行逻辑（Implementation）**：实际的函数/API 调用
5. **输出格式（Output Schema）**：返回结果的结构定义
6. **权限与约束（Permission）**：是否需要用户确认、资源限制

**设计原则**：
- 单一职责：一个 Skill 只做一件事
- 描述清晰：帮助 Agent 在规划时准确选择
- 错误处理：返回标准化错误信息，便于 Agent 感知失败

### Q6：你了解 Harness 工程吗？主要难点是什么？

**答案（源自蚂蚁 Agent 优化算法面试）：**

Harness 工程指**将 AI Agent 能力安全、可靠地集成到现有业务系统中的工程化实践**。

**主要难点**：
1. **安全边界设计**：Agent 的自主行动可能造成不可逆操作（删除数据、发送错误邮件），需要分级权限、人工确认兜底。
2. **可靠性保障**：LLM 输出的不确定性导致失败率高，需要重试机制、降级策略。
3. **状态管理**：复杂任务跨多轮对话，状态持久化与恢复。
4. **性能与成本**：多次 LLM 调用导致延迟和 token 消耗，需要缓存、压缩上下文。
5. **可观测性**：Agent 决策过程黑盒，需记录完整轨迹用于调试和审计。

## 二、记忆系统

### Q7：短期记忆和长期记忆如何设计？如何协同工作？

**答案：**

**短期记忆**：
- 作用域：单次会话内的对话历史
- 实现方式：滑动窗口保留最近 N 轮对话；或对历史进行**摘要压缩**，用摘要 + 最近几轮替代完整历史
- 存储位置：内存 / Redis（带 TTL）

**长期记忆**：
- 作用域：跨会话持久化信息
- 实现方式：
  1. **向量化存储**：将用户重要信息、偏好、事实提取后生成 Embedding 存入向量数据库（如 Milvus）
  2. **结构化存储**：关系型数据库记录用户画像
  3. **图数据库**：记录实体间关系（如 Dendritic Network）
- 检索方式：在每次对话开始时，从长期记忆中检索相关事实注入上下文

**协同流程**：
1. 新对话开始 → 从长期记忆中检索与当前意图相关的历史信息
2. 注入短期记忆窗口
3. 对话过程中，识别重要新信息 → 实时更新长期记忆
4. 会话结束 → 短期记忆压缩为摘要，写入长期记忆

---

### Q8：大量记忆如何处理？Token 限制下的记忆压缩策略有哪些？

**答案：**

**核心矛盾**：LLM 上下文窗口有限（如 128K），但对话历史可能无限增长，记忆的存储量（向量库）也可能膨胀到几百万条。

**压缩策略**：
1. **滑动窗口截断**：只保留最近 K 轮对话，超出部分直接丢弃（最简单但丢信息）
2. **摘要压缩**：用 LLM 将历史对话压缩为简洁摘要，摘要 + 最近几轮形成上下文
3. **分层索引 + 检索式召回**：所有历史对话存入向量库，每轮新对话时根据语义检索最相关的历史片段，而非全部保留在上下文（类似 RAG 思想）
4. **层次化记忆**：将记忆分为“核心事实”（重要不丢）、“情景记忆”（检索召回）、“过程记忆”（摘要保留）三层

**大量记忆处理**：
- 使用向量数据库 + 高效索引（HNSW、IVF）
- 设置记忆淘汰策略（LRU、重要性评分）
- 对象存储归档冷记忆，热记忆放 Redis

---

### Q9：为什么说 Hermès 的记忆系统比 OpenClaw 更好？（对比题）

**答案：**

**Hermès 的优势**：
1. **自我进化机制**：Hermès 能根据用户反馈和使用模式自动优化记忆的存储与检索策略。例如，如果某类信息经常被访问，会自动提升其重要性权重。
2. **结构化记忆图谱**：采用图结构存储实体关系，而非单纯向量检索，推理能力更强。
3. **主动遗忘策略**：基于信息熵判断记忆的长期价值，低价值信息会被降权或归档，避免噪声积累。
4. **情境感知检索**：不只做相似度匹配，还结合当前对话场景、用户情绪等多信号融合检索。

**OpenClaw 的局限**：
- 记忆模块相对通用，缺乏自动优化能力
- 主要依赖向量相似度检索，对复杂关联推理支持不足

（注：具体实现细节可能因版本而异，面试时以阐述对比思路为主。）

## 三、RAG 检索增强生成

### Q10：详细说明 RAG 的完整流程，并画出链路图。

**答案：**

RAG（Retrieval-Augmented Generation）是解决 LLM 知识过时和幻觉的核心方案，流程分为**离线索引**和**在线检索增强**两阶段。

**链路图**：
[离线阶段]
文档 → 解析切块 → Embedding生成 → 向量数据库存储

[在线阶段]
用户Query → Embedding生成 → 向量检索（Top-K） → 重排序 → 拼接Prompt → LLM生成 → 答案

**详细步骤**：
1. **文档处理**：PDF/网页/数据库 → 解析为纯文本 → 按语义切块（Chunk），通常 512-1024 token
2. **向量化**：使用 Embedding 模型（BGE、E5、text-embedding-3）将每个块转为向量
3. **存储索引**：存入向量数据库（Milvus、Pinecone、Weaviate），建立 HNSW 等索引
4. **用户查询**：输入问题，用同一 Embedding 模型生成查询向量
5. **检索**：在向量库中执行 ANN 搜索，返回 Top-K 相关文档块
6. **重排序**（可选）：用 Reranker 模型对 Top-K 结果精排，选出最相关的 Top-N
7. **增强生成**：将检索到的文档块与用户问题拼接成 Prompt，送往 LLM 生成答案

---

### Q11：RAG 如何优化？分别从 Embedding 层和检索层说明。

**答案：**

**Embedding 层优化**：
1. **模型选型**：选用领域适配的 Embedding 模型（如 BGE-large、E5-mistral），或使用通用模型但进行**领域微调**
2. **Chunk 切分策略**：
   - 固定长度 vs 语义切分（按段落、句子）
   - 重叠窗口（Overlap）避免信息断裂
   - 多粒度索引（文档级摘要 + 块级细节）
3. **查询改写**：对用户问题进行扩展、纠错、多角度重写，提高召回覆盖率
4. **假设性文档嵌入（HyDE）**：用 LLM 先生成一个假设答案，再用该答案向量去检索，缩小语义鸿沟

**检索层优化**：
1. **混合检索**：结合**语义检索**（向量相似度）和**关键词检索**（BM25），兼顾语义和精确匹配
2. **动态权重**：根据查询类型调整混合权重（实体查询偏 BM25，分析查询偏语义）
3. **重排序（Rerank）**：用专用 Reranker 模型（如 BGE-Reranker、Cohere Rerank）对召回结果二次排序
4. **多路召回融合**：从不同索引（标题索引、段落索引、知识图谱）分别召回后合并去重
5. **缓存策略**：热门查询结果缓存到 Redis，定期更新

### Q12：如何评估 RAG 系统的效果？指标有哪些？

**答案：**

RAG 评估分**检索质量**和**生成质量**两个维度。

**检索质量指标**：
- **命中率（Hit Rate）**：Top-K 结果中是否包含正确答案
- **MRR（Mean Reciprocal Rank）**：第一个正确答案排名的倒数均值
- **NDCG**：考虑排序位置的归一化折损累计增益
- **精确率/召回率**：检索出的相关文档比例

**生成质量指标**：
- **答案正确率**：答案是否与参考答案一致（需人工或 LLM-as-judge 评估）
- **忠实度（Faithfulness）**：生成内容是否完全基于检索到的文档，无幻觉
- **相关性（Relevance）**：答案是否直接回答了问题
- **ROUGE / BLEU**：与参考答案的词汇重叠度（仅作参考）

**实用评估方法**：
- 构建高质量测试集（问答对 + 参考答案 + 相关文档标注）
- 使用 RAGAS 框架自动化评估
- 人工抽检 + 用户反馈闭环

### Q13：RAG 知识库体量有多大？如何应对大规模检索的性能瓶颈？

**答案：**

**体量参考**：生产环境中知识库可达 10 万+ 文档、百万级 Chunk，单次检索需在百毫秒内完成。

**性能优化措施**：
1. **索引优化**：选用高效 ANN 索引（HNSW、IVF-PQ），调节 efSearch 参数平衡速度和精度
2. **分片与分区**：按主题、时间分库分表，缩小检索范围
3. **查询过滤**：先通过元数据过滤（如“只检索近一年文档”），再向量检索
4. **结果缓存**：热点查询结果存入 Redis，命中率可达 60%
5. **异步并发**：检索与生成解耦，检索可以并行查询多个分片
6. **模型小型化**：使用轻量级 Embedding 模型，甚至本地部署加速
7. **近似计算**：在可接受精度损失下使用量化索引（PQ、SQ）

## 四、LangChain / LangGraph 框架

### Q14：LangChain 的核心组件有哪些？各起什么作用？

**答案：**

LangChain 是 LLM 应用开发框架，核心组件如下：

| 组件 | 作用 |
|------|------|
| **Models** | 封装 LLM（ChatOpenAI等）和 Embedding 模型 |
| **Prompts** | 提示模板管理，支持变量注入、Few-shot |
| **Chains** | 将多个组件串联为执行流程 |
| **Tools** | 定义外部工具接口，Agent 可调用 |
| **Memory** | 对话历史存储与检索，支持多种后端 |
| **Retrievers** | 从向量数据库等后端检索文档 |
| **Agents** | 使用 LLM 进行规划和工具调用的智能体 |

此外还有 **Indexes**（文档索引加载器）、**Callbacks**（日志、监控钩子）等辅助模块。

---

### Q15：LangChain 和 LangGraph 有什么区别？分别在什么场景下使用？

**答案：**

| 维度 | LangChain | LangGraph |
|------|-----------|-----------|
| 设计理念 | 链式流程（Chain） | 图结构状态机 |
| 流程控制 | 线性 / 预设分支 | 灵活的循环、条件、并行 |
| 状态管理 | 简单内存传递 | 内置状态图，跨节点持久化 |
| 适合场景 | 简单 RAG、单步 Agent | 多 Agent 协作、复杂工作流、带人类介入的流程 |

**使用建议**：
- 简单问答、单步工具调用 → LangChain Chain
- 需要循环推理、多个 Agent 协同、人工审核介入 → LangGraph

**示例**：一个“客服工单处理系统”需要“判断问题类型 → 查知识库 → 不行问专家 → 专家处理 → 总结归档”这种带分支和循环的流程，就适合 LangGraph 实现。

---

### Q16：什么是 LCEL？简述其作用。

**答案：**

LCEL（LangChain Expression Language）是 LangChain 推出的**声明式链式表达语法**，使用 `|` 管道操作符串联组件。

```python
from langchain_core.runnables import RunnablePassthrough

chain = (
    {"context": retriever, "question": RunnablePassthrough()}
    | prompt
    | llm
    | output_parser
)
```

**优点：**

- 代码简洁，逻辑清晰
- 自动支持异步、流式输出、并行执行
- 集成了 LangSmith 追踪、缓存等能力

## 五、Prompt Engineering 与 AI 编程工具

### Q17：你用过哪些 AI 编程工具？怎么用的？Claude Code 在工作流中起到什么作用？

**答案：**

工具经验：Claude Code、Cursor、GitHub Copilot、Trae 等。

**工作流：**

- **需求分析**：用 Claude Code / Cursor Chat 讨论技术方案
- **代码生成**：描述功能需求，由 AI 生成初始代码
- **代码审查（CR）**：对 AI 生成的代码逐段审查，重点检查边界条件、错误处理、性能隐患
- **调试**：将报错信息粘贴给 AI，请求修复建议
- **测试生成**：让 AI 为函数生成单元测试
- **文档撰写**：AI 辅助生成接口文档、README

**Claude Code 的优势（来自实际使用）：**

- **终端原生集成**，直接操作文件系统
- **上下文理解能力强**，能基于整个项目文件做修改建议
- **支持自定义指令**，可配置代码风格规范

**CR 流程示例（设计记忆模块时）：**

AI 生成了向量存入代码，审查后发现未处理 Embedding 失败重试、未设置唯一约束导致重复记忆、未考虑旧记忆更新而非新增。逐一修正后提交。

### Q18：用 Claude Code 生成代码失败或逻辑错误时，你怎么进行 Code Review？

**答案：**

**CR 重点：**

- **业务逻辑正确性**：检查是否满足需求，有无遗漏边缘情况
- **异常处理**：网络超时、API 调用失败、空值等是否有处理
- **性能**：循环内是否有不必要的 I/O、大列表操作
- **安全性**：用户输入是否合法，SQL 注入、权限校验
- **代码可维护性**：命名、注释、模块划分是否清晰

**具体流程：**

- 将 AI 生成的代码粘贴到 IDE 中
- 逐行阅读，标注可疑处
- 针对关键模块自己编写测试用例验证
- 若逻辑错误，将错误现象反馈给 AI 请求修正，或者手动修改
- 提交前再次同 AI 讨论，确认改动无副作用

## 六、模型微调与性能优化（加分项）

### Q19：LoRA、QLoRA、P-Tuning 的核心区别是什么？

**答案：**

| 方法 | 核心思想 | 可训练参数量 | 特点 |
|------|----------|-------------|------|
| LoRA | 在权重矩阵旁并联低秩分解矩阵，只训练低秩矩阵 | 极低（<1%） | 推理时可合并到原权重，无额外延迟 |
| QLoRA | LoRA + 4bit 量化基础模型 | 更低 | 单卡 24G 显存可微调 70B 模型 |
| P-Tuning | 在输入层加入可学习的连续 Prompt 向量，不修改模型权重 | 极少 | 适合 NLU 任务，不改变生成能力 |

**适用场景：**

- **LoRA**：主流选择，通用性好
- **QLoRA**：资源受限环境
- **P-Tuning**：快速适应特定任务，如分类、抽取

### Q20：如何进行 Agent 后训练？需要哪些数据？

**答案：**

**训练方法：**

- **Tool-Use 微调**：构造大量"用户指令→工具调用→观察→最终回答"的对话数据，使用 SFT（监督微调）训练模型按格式输出工具调用
- **RLHF/DPO 对齐**：收集人类对 Agent 轨迹的偏好数据，通过强化学习或 DPO 优化 Agent 决策策略
- **轨迹蒸馏**：用强模型（如 GPT-4）生成 Agent 交互轨迹，蒸馏到小模型

**数据构造：**

- 收集真实用户使用日志，清洗后标注
- 使用 Self-Instruct 方法让强模型生成多样化指令和对应的工具调用链
- 注意包含失败恢复路径，提高鲁棒性

## 七、Agent 前沿与趋势

### Q21：最近的 Agent 相比年初有什么进展和变化？（2026 年视角）

**答案：**

- **架构统一化**：MCP（Model Context Protocol）成为工具连接的事实标准，不同 Agent 间的工具可复用
- **多模态 Agent**：从纯文本扩展到视觉、语音的多模态输入和操作
- **端侧 Agent**：小模型（如 Phi、Gemma）结合量化技术，Agent 可在手机、IoT 设备本地运行
- **面向过程的 Agent 评估**：从只看最终结果到评估整个推理-行动轨迹的正确性、效率、安全性
- **企业级 Harness 工程成熟**：出现更多 AgentOps 平台（如 LangSmith、Weights & Biases），提供可观测性、安全护栏、成本控制
- **Agent 间协作协议**：多 Agent 系统开始使用标准通信语言（如 A2A Protocol）协同工作

### Q22：太长的上下文如何处理？有哪些工程方案？

**答案：**

对 Agent 而言，长上下文可能来自长文档、长对话历史、大量工具返回结果。

**处理方案：**

- **分块检索**：不要把所有内容都扔进上下文，用 RAG 思维只检索相关片段
- **结构化摘要**：长文档先自动生成层级大纲和摘要，Agent 只读摘要，需要细节时再定向检索
- **记忆压缩**：将历史对话实时压缩为关键状态变量（如用户目标、当前步骤、已获信息），而非保留全文
- **模型选择**：使用支持超长上下文（1M+）的模型（如 Gemini 2.0），但注意注意力衰减问题——中间的信息容易被忽略，应把关键信息放在开头或结尾
- **流式处理**：对于大体积工具输出，边接收边分析，只提取关键字段

### Q23：如何设计一个可自我进化的 Agent（参考 Hermès）？

**答案：**

自我进化机制的核心是"从交互中学习并自动调整策略"。

**组件设计：**

- **反馈收集器**：自动记录用户的显式反馈（👍/👎）和隐式信号（重复提问、中断对话、任务完成时长）
- **经验提取器**：定期对反馈日志做分析，提取成功/失败的模式（如"当用户语气急促时，缩短回答更有效"）
- **策略更新器**：
  - 调整记忆检索的权重、摘要策略
  - 优化 Prompt 模板（A/B 测试）
  - 更新工具选择的优先级规则
- **安全回滚**：新策略先在影子模式运行，对比旧策略若无显著提升则回滚

**面试亮点**：可类比推荐系统的在线学习，形成"交互→反馈→策略更新→更好交互"的正循环。
