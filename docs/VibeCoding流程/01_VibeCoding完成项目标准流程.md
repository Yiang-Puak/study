# Vibe Coding 完成项目的标准流程

## 1. 什么是 Vibe Coding

Vibe Coding 不是“随便让 AI 写代码”，而是一种以人类目标为中心、以 AI Agent 为执行伙伴的项目开发方式。

它的核心思想是：

- **人负责方向**：明确目标、判断取舍、验收结果。
- **AI 负责加速**：辅助分析需求、生成方案、编写代码、解释错误、生成文档。
- **流程负责质量**：通过需求文档、任务拆解、迭代文档、Prompt 规范和验收标准，避免项目失控。

一个成熟的 Vibe Coding 流程不是直接说“帮我做一个系统”，而是按照以下路径推进：

```text
想法 → 需求澄清 → 需求文档 → 技术方案 → 任务拆解 → 迭代开发 → 测试验收 → 复盘优化 → 部署交付
```

---

## 2. Vibe Coding 标准项目流程总览

推荐流程如下：

| 阶段 | 目标 | 主要产物 |
|---|---|---|
| 1. 项目构思 | 明确要解决什么问题 | 项目一句话描述 |
| 2. 需求澄清 | 把模糊想法变成清晰需求 | 需求问答记录 |
| 3. 需求文档 | 固化项目范围和功能边界 | PRD / 需求文档 |
| 4. 技术方案 | 确定技术架构和实现路径 | 技术设计文档 |
| 5. 任务拆解 | 把大项目拆成可执行任务 | 任务列表 / 迭代计划 |
| 6. Prompt 设计 | 让 Agent 准确理解上下文 | Agent Prompt / 规则文档 |
| 7. 迭代开发 | 分阶段实现功能 | 代码、提交记录、迭代文档 |
| 8. 测试验收 | 验证是否满足需求 | 测试用例、验收清单 |
| 9. 部署交付 | 让项目可被真实使用 | 部署文档、使用说明 |
| 10. 复盘优化 | 总结问题并持续改进 | 复盘文档、下一轮迭代计划 |

## 3. 第一步：如何分析需求

需求分析的目标是把“我想做一个东西”变成“我要为谁，在什么场景下，解决什么问题”。

### 3.1 需求分析的核心问题

你可以从以下几个问题开始：

#### 1. 用户是谁

需要明确目标用户，而不是泛泛地说“所有人”。

示例：

```text
错误：我要做一个数据分析系统。
正确：我要做一个给中小企业运营人员使用的数据分析助手。
```

需要回答：

- **目标用户**：谁会使用这个系统？
- **用户水平**：他们是技术人员还是非技术人员？
- **使用频率**：每天用、每周用，还是偶尔用？
- **使用环境**：网页端、移动端、本地工具、企业内网？

#### 2. 痛点是什么

痛点必须具体。

示例：

```text
错误：用户分析数据很麻烦。
正确：运营人员不会写 SQL，每次想看销售趋势都要找数据分析师，等待时间长。
```

需要回答：

- **现在怎么做**：用户当前如何解决这个问题？
- **哪里低效**：时间成本、学习成本、沟通成本在哪里？
- **为什么需要新系统**：现有方式为什么不够好？

#### 3. 目标是什么

目标最好能量化。

示例：

```text
让运营人员通过自然语言提问，在 10 秒内生成销售趋势图表。
```

可以从以下角度描述：

- **效率目标**：减少多少操作步骤？
- **体验目标**：用户是否无需学习复杂工具？
- **业务目标**：提升查询效率、降低人工成本、提高转化率？

#### 4. 核心功能是什么

不要一开始就堆很多功能。先找 MVP，也就是最小可用版本。

示例：

```text
MVP 功能：
1. 上传 CSV 文件
2. 用户用自然语言提问
3. 系统生成分析结果
4. 系统展示图表
```

#### 5. 什么暂时不做

明确“不做什么”非常重要，可以防止需求膨胀。

示例：

```text
第一版不做：
1. 用户权限系统
2. 多租户企业管理
3. 实时数据库同步
4. 复杂 BI 大屏编辑器
```

## 4. 用 LLM 辅助生成和澄清需求

你不应该只让 LLM “帮我写需求文档”，而应该让它先向你提问。

### 4.1 需求澄清 Prompt

可以使用下面的 Prompt：

```text
你是一个资深产品经理，请帮助我澄清一个项目需求。

项目初始想法：
【在这里写你的项目想法】

请你不要急着生成完整需求文档，而是先从以下维度向我提问：
1. 目标用户
2. 使用场景
3. 用户痛点
4. 核心功能
5. 非核心功能
6. 数据来源
7. 交互方式
8. 权限与安全
9. 部署环境
10. 成功指标

要求：
- 每次最多问 10 个关键问题。
- 问题要具体，不要空泛。
- 如果我的描述模糊，请指出模糊点。
- 最后请根据我的回答总结一版 MVP 范围。
```

### 4.2 让 LLM 反向挑战需求

很多需求看起来合理，但实现成本高或价值不清晰。可以让 LLM 帮你做反向审查。

```text
你是一个严格的产品评审专家，请审查下面的项目需求。

需求内容：
【粘贴需求草稿】

请从以下角度指出问题：
1. 哪些需求不够清晰？
2. 哪些功能可能超出 MVP 范围？
3. 哪些地方存在技术实现风险？
4. 哪些用户场景没有被覆盖？
5. 哪些功能可以延后？
6. 如果只能用两周完成，应该保留哪些功能？

请输出：
- 问题清单
- 修改建议
- MVP 推荐范围
- 后续迭代建议
```

### 4.3 让 LLM 生成用户故事

用户故事适合把需求变成开发任务。

```text
请根据下面的项目需求，生成用户故事。

项目需求：
【粘贴需求文档】

格式要求：
作为【某类用户】，我希望【完成某个动作】，以便【获得某种价值】。

请按模块分类输出，并为每个用户故事补充：
1. 优先级：P0 / P1 / P2
2. 验收标准
3. 边界情况
```

## 5. 需求文档模板

下面是一份适合 Vibe Coding 的需求文档模板。

```markdown
# 项目需求文档

## 1. 项目背景

说明为什么要做这个项目，当前存在什么问题。

## 2. 项目目标

说明项目希望达成什么结果。

示例：
- 让用户可以通过自然语言查询数据。
- 将人工分析时间从 30 分钟降低到 1 分钟。

## 3. 目标用户

描述核心用户群体。

| 用户类型 | 特征 | 主要需求 |
|---|---|---|
| 运营人员 | 不懂 SQL | 快速查询业务数据 |
| 管理者 | 关注结果 | 查看趋势和结论 |

## 4. 使用场景

描述用户在什么情况下使用系统。

### 场景 1：日常数据查询

用户想知道最近 7 天销售额变化趋势。

### 场景 2：异常问题分析

用户发现某天订单量下降，希望快速定位原因。

## 5. 核心功能

| 功能模块 | 功能描述 | 优先级 | 是否 MVP |
|---|---|---|---|
| 数据上传 | 支持上传 CSV 文件 | P0 | 是 |
| 自然语言问答 | 用户可以输入问题 | P0 | 是 |
| 图表展示 | 展示折线图、柱状图 | P0 | 是 |
| 用户登录 | 账号密码登录 | P1 | 否 |

## 6. 非功能需求

包括性能、安全、兼容性、可维护性等。

示例：
- 页面响应时间小于 3 秒。
- 单个 CSV 文件最大支持 20MB。
- API Key 不允许写死在前端代码中。

## 7. 不做什么

明确第一版不包含的内容。

示例：
- 不做多租户管理。
- 不做复杂权限系统。
- 不接入实时数据库。

## 8. 数据需求

说明数据从哪里来，格式是什么。

| 数据源 | 数据格式 | 用途 |
|---|---|---|
| 用户上传 | CSV | 数据分析 |
| 本地数据库 | SQLite | 保存历史记录 |

## 9. 交互流程

用文字描述用户操作路径。

```text
用户进入首页 → 上传文件 → 输入问题 → 系统分析 → 展示结论和图表
```

## 10. 验收标准

定义什么叫“完成”。

示例：
- 用户可以成功上传 CSV。
- 用户输入“统计每月销售额”，系统能返回正确图表。
- 当文件格式错误时，系统能给出明确错误提示。

## 11. 迭代计划

| 版本 | 目标 | 功能 |
|---|---|---|
| v0.1 | MVP 验证 | 上传、问答、图表 |
| v0.2 | 体验优化 | 历史记录、错误提示 |
| v0.3 | 权限增强 | 登录、用户数据隔离 |
```

## 6. 如何拆解任务

需求文档完成后，不要直接让 Agent 写完整项目。你应该先让它拆任务。

### 6.1 任务拆解 Prompt

```text
你是一个资深技术负责人，请根据下面的需求文档，将项目拆解成可执行开发任务。

需求文档：
【粘贴需求文档】

请输出：
1. 模块划分
2. 每个模块的开发任务
3. 每个任务的输入、输出、依赖关系
4. 推荐开发顺序
5. 每个任务的验收标准
6. 哪些任务适合先做 MVP
7. 哪些任务可以延后

要求：
- 任务粒度不要太大，每个任务最好能在 0.5 到 2 天内完成。
- 不要只写功能名，要写清楚要改哪些部分。
- 如果有不确定的地方，请列出问题。
```

### 6.2 推荐的任务拆解结构

```markdown
# 任务拆解文档

## 1. 模块划分

- 前端页面
- 后端 API
- 数据库
- AI Agent
- 文件处理
- 测试与部署

## 2. 任务列表

| ID | 任务 | 优先级 | 依赖 | 验收标准 |
|---|---|---|---|---|
| T001 | 初始化项目结构 | P0 | 无 | 项目可以正常启动 |
| T002 | 实现文件上传接口 | P0 | T001 | 可以上传 CSV 并返回字段信息 |
| T003 | 实现自然语言分析接口 | P0 | T002 | 输入问题后返回分析结果 |
| T004 | 实现图表展示页面 | P0 | T003 | 前端能展示后端返回的图表数据 |

## 3. 开发顺序

1. 初始化项目
2. 打通前后端基础接口
3. 实现核心业务逻辑
4. 接入 AI 能力
5. 增加错误处理
6. 完善界面和测试
```

## 7. 如何进行迭代开发

Vibe Coding 最容易失败的原因是一次性让 Agent 写太多代码。正确方式是小步迭代。

### 7.1 每轮迭代的标准流程

```text
选择一个小任务 → 描述上下文 → 让 Agent 制定方案 → 审查方案 → 执行修改 → 运行测试 → 记录结果
```

### 7.2 单次迭代建议

每次只让 Agent 完成一个明确目标。

#### 好的任务描述

```text
请在现有 FastAPI 后端中新增一个 CSV 文件上传接口。
要求：
1. 接口路径为 /api/upload
2. 只接受 .csv 文件
3. 上传后读取字段名和前 5 行数据
4. 返回 JSON，包括 columns 和 preview
5. 如果文件格式不正确，返回明确错误信息
6. 不要修改前端代码
```

#### 不好的任务描述

```text
帮我把整个数据分析系统做完。
```

### 7.3 迭代中的关键原则

- **一次只做一件事**：避免大范围失控修改。
- **先让 Agent 读代码**：不要让它凭空假设项目结构。
- **先方案后代码**：复杂任务先让 Agent 说明修改计划。
- **保留验收标准**：每次任务都要知道如何判断完成。
- **及时运行测试**：不要堆到最后才测试。
- **记录变更原因**：方便后续维护和回滚。

## 8. 迭代文档模板

```markdown
# 迭代文档：v0.1

## 1. 迭代目标

本轮迭代要完成什么。

示例：
完成 MVP 的文件上传、数据预览和自然语言问答基础链路。

## 2. 本轮范围

### 包含

## 3. 任务列表

| 任务 ID | 任务名称 | 状态 | 负责人 | 验收标准 |

| T001 | 初始化项目结构 | 未开始 | Agent | 项目能启动 |
| T002 | 文件上传接口 | 未开始 | Agent | 上传 CSV 后返回预览数据 |
| T003 | 前端上传页面 | 未开始 | Agent | 用户能选择文件并上传 |

## 4. 每日记录

### Day 1

- 完成：
- 问题：
- 下一步：

### Day 2

- 完成：
- 问题：
- 下一步：

## 5. 测试记录

| 测试项 | 输入 | 预期结果 | 实际结果 | 是否通过 |
|---|---|---|---|---|

## 6. 遗留问题

- 问题 1：
- 问题 2：

## 7. 下一轮计划

- 增加历史记录
- 优化错误提示
- 增加图表展示

## 9. Prompt 工程：如何让 Agent 真正理解项目

Agent 的表现很大程度取决于你给它的上下文。好的 Prompt 应该包含：

- **角色**：你希望 Agent 以什么身份工作。
- **背景**：项目是什么，当前阶段是什么。
- **目标**：这次要完成什么。
- **约束**：不能改什么，必须遵守什么。
- **输入**：需求文档、代码结构、错误日志等。
- **输出格式**：你希望它输出方案、代码、表格还是清单。
- **验收标准**：什么结果算完成。

### 9.1 通用 Agent 开发 Prompt 模板

```text
你是一个资深全栈工程师，现在要在一个已有项目中完成开发任务。

项目背景：
【简要描述项目是做什么的】

当前技术栈：
【例如：React + FastAPI + SQLite + LangChain】

当前任务：
【明确写一个任务，不要写多个大任务】

需求细节：
1. 【要求 1】
2. 【要求 2】
3. 【要求 3】

约束条件：
1. 不要修改与本任务无关的文件。
2. 不要删除已有功能。
3. 不要硬编码 API Key。
4. 如果需要新增依赖，请先说明原因。
5. 修改前请先阅读相关文件并说明计划。

验收标准：
1. 【如何验证功能完成】
2. 【错误情况如何处理】
3. 【是否需要测试】

请按照以下步骤执行：
1. 先分析现有项目结构。
2. 找到相关文件。
3. 给出简短修改计划。
4. 在确认合理后进行代码修改。
5. 修改后说明改了什么，以及如何测试。
```

### 9.2 让 Agent 先读代码的 Prompt

```text
请先不要修改代码。

请阅读当前项目中与【功能名称】相关的文件，并回答：
1. 相关入口文件在哪里？
2. 数据流是怎样的？
3. 哪些文件可能需要修改？
4. 当前实现有什么限制？
5. 你建议的最小修改方案是什么？

在我确认之前，不要进行代码修改。
```

### 9.3 让 Agent 修改代码的 Prompt

```text
请根据你刚才的方案进行实现。

要求：
1. 只修改与本任务直接相关的文件。
2. 保持现有代码风格。
3. 不要删除已有注释和功能。
4. 如果发现需求与现有代码冲突，先停止并说明问题。
5. 修改完成后给出测试方法。
```

### 9.4 让 Agent Debug 的 Prompt

```text
当前项目出现如下错误：

错误日志：
【粘贴完整错误日志】

复现步骤：
1. 【步骤 1】
2. 【步骤 2】
3. 【步骤 3】

请你按照以下方式排查：
1. 先解释错误含义。
2. 判断最可能的根因。
3. 指出需要查看哪些文件。
4. 不要直接大范围重构。
5. 优先给出最小修复方案。
6. 修复后说明如何验证。
```

### 9.5 让 Agent 写测试的 Prompt

```text
请为下面的功能补充测试。

功能描述：
【描述功能】

已有代码位置：
【文件路径】

测试要求：
1. 覆盖正常输入。
2. 覆盖异常输入。
3. 覆盖边界情况。
4. 测试命名要清晰。
5. 不要为了测试而修改业务逻辑，除非发现代码不可测试，请先说明。
```

## 11. 项目开发中的常见坑

### 11.1 需求太大

错误方式：

```text
帮我做一个类似 Notion + ChatGPT + 飞书多维表格的系统。
```

正确方式：

```text
先做一个最小版本：用户可以创建一篇笔记，并使用 AI 总结这篇笔记。
```

### 11.2 上下文不足

Agent 不知道你的项目结构、技术栈和已有代码时，容易乱改。

解决方法：

```text
请先阅读项目结构和相关文件，不要马上修改代码。
```

### 11.3 一次修改太多

一次性让 Agent 修改十几个文件，很容易引入隐藏 bug。

解决方法：

- 每次只做一个模块。
- 每次修改后运行测试。
- 每轮迭代都记录变更。

### 11.4 没有验收标准

如果没有验收标准，Agent 做完后你也不知道是否真的完成。

解决方法：

```text
请为每个功能补充可执行的验收标准。
```

### 11.5 让 Agent 自由发挥 UI

如果你对 UI 有要求，需要明确风格。

示例：

```text
UI 风格要求：
1. 简洁现代。
2. 使用卡片式布局。
3. 主色调为蓝紫渐变。
4. 移动端优先。
5. 页面需要有空状态、加载状态和错误状态。
```

## 12. 一个完整示例：从想法到开发

### 12.1 初始想法

```text
我想做一个 AI 简历优化助手。
```

### 12.2 需求澄清后

```text
目标用户：正在找实习或校招的学生。
核心痛点：不知道简历哪里有问题，也不知道如何针对岗位修改。
MVP 功能：
1. 上传简历文本。
2. 输入目标岗位 JD。
3. AI 分析简历与岗位的匹配度。
4. 给出修改建议。
5. 生成优化后的简历片段。
第一版不做：
1. 在线简历编辑器。
2. 多模板导出。
3. 账号系统。
```

### 12.3 MVP 需求

```text
用户上传简历和岗位 JD 后，系统调用 LLM 分析两者匹配度，输出优势、短板、关键词缺失和修改建议。
```

### 12.4 第一次迭代任务

```text
实现一个命令行版本：
1. 用户输入简历文本文件路径。
2. 用户输入 JD 文本文件路径。
3. 程序读取两个文件。
4. 调用 LLM 生成匹配分析。
5. 将结果保存为 Markdown 文件。
```

### 12.5 给 Agent 的开发 Prompt

```text
你是一个资深 Python 工程师。

我要开发一个 AI 简历优化助手的 MVP 命令行版本。

当前任务：
实现一个 Python CLI 程序，读取简历文件和岗位 JD 文件，并调用 LLM 生成分析报告。

要求：
1. 使用 Python 实现。
2. 支持命令行参数 --resume 和 --jd。
3. 输入文件为 txt 或 md。
4. 输出 Markdown 分析报告。
5. API Key 从环境变量读取，不能硬编码。
6. 如果文件不存在，要给出清晰错误提示。
7. 代码结构要清晰，便于后续改成 Web 版本。

请先给出项目结构和实现计划，不要马上写代码。
```

## 13. 推荐的 Vibe Coding 工作流

### 13.1 新项目工作流

```text
1. 写一句话项目想法
2. 让 LLM 提问澄清需求
3. 回答问题
4. 让 LLM 生成 MVP 范围
5. 让 LLM 生成需求文档
6. 让 LLM 审查需求文档
7. 让 LLM 生成技术方案
8. 让 LLM 拆解任务
9. 每次只让 Agent 完成一个任务
10. 每轮结束写迭代记录
```

### 13.2 已有项目工作流

```text
1. 让 Agent 读取项目结构
2. 让 Agent 找到相关模块
3. 让 Agent 解释当前实现
4. 说明你要改的需求
5. 让 Agent 给出最小修改方案
6. 确认方案后再让 Agent 修改
7. 修改后运行测试
8. 记录本次变更
```

### 13.3 Debug 工作流

```text
1. 粘贴完整错误日志
2. 说明复现步骤
3. 说明期望结果和实际结果
4. 让 Agent 判断根因
5. 让 Agent 找相关代码
6. 让 Agent 给出最小修复方案
7. 修复后重新运行测试
8. 记录问题原因和解决方案
```

## 15. 高质量 Prompt 的公式

可以记住这个公式：

```text
角色 + 背景 + 目标 + 输入 + 约束 + 步骤 + 输出格式 + 验收标准
```

示例：

```text
你是一个资深后端工程师。

背景：这是一个 FastAPI 项目，已经有用户模块和数据库连接。

目标：新增一个订单查询接口。

输入：用户传入 order_id。

约束：
1. 不要修改用户模块。
2. 不要新增数据库表。
3. 复用已有数据库连接。

步骤：
1. 先阅读项目结构。
2. 找到 API 路由注册方式。
3. 给出修改计划。
4. 再实现代码。

输出格式：
1. 修改文件列表。
2. 功能说明。
3. 测试方法。

验收标准：
1. GET /api/orders/{order_id} 能返回订单详情。
2. order_id 不存在时返回 404。
3. 参数非法时返回 400。
```

## 16. 最终总结

Vibe Coding 的关键不是“让 AI 一次性把项目写完”，而是建立一套稳定的人机协作流程。

你需要做到：

- **先需求，后代码**：没有清晰需求，不要直接开发。
- **先文档，后实现**：需求文档和任务拆解能显著降低返工。
- **先小步，后扩展**：每次只完成一个可验证的小目标。
- **先验收，后迭代**：每轮都要有测试和验收标准。
- **先约束，后生成**：Prompt 里必须写清楚边界和限制。

最推荐的执行顺序：

```text
需求澄清 Prompt → 需求文档 → 需求评审 Prompt → 任务拆解 → 单任务开发 Prompt → 测试验收 → 迭代记录
```

只要你坚持这个流程，Vibe Coding 就不再是“碰运气让 AI 写代码”，而会变成一套可复制、可复盘、可持续迭代的项目开发方法。
