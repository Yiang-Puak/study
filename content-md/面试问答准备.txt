一、智能数据查询助手 DataInsight Agent
########################################

Q1: 请简单介绍一下你的 DataInsight Agent 项目？
A1: 这是一个面向非技术用户（如业务运营、销售主管）的智能数据查询助手。用户输入自然语言问题，系统自动完成意图识别、SQL 生成、安全校验、SQL 执行，最后以文字业务解读 + ECharts 可视化图表的形式返回结果。
技术栈是 Python + FastAPI（后端）+ Streamlit（前端）+ LangGraph（Agent 框架）+ SQLite（数据库）+ 阿里云百炼 Qwen（大模型）。

Q2: 为什么选择 LangGraph 而不是直接用 LangChain 的 AgentExecutor？
A2: LangGraph 基于状态机模型，每个节点职责明确，流程可控：
- 可以精确定义节点间的流转逻辑和条件边，比如 SQL 执行失败后重试、闲聊直接返回
- 支持条件分支（conditional edges），比如根据 intent 决定走数据查询还是闲聊拒绝
- 状态通过 TypedDict 统一管理，各节点之间数据传递清晰
而 AgentExecutor 更适合工具调用型的场景，对于这种"固定流程 + 条件分支"的 Agent，LangGraph 更合适。

Q3: 你的 Agent 状态机有哪些节点？流程是怎样的？
A3: v1.1 版本共 5 个节点：
1. schema_rag_node：返回数据库表结构（DDL 格式，带 COMMENT 注释）
2. intent_and_sql_node：调用 LLM 做意图识别 + SQL 生成，JSON Mode 输出
3. sql_execute_node：执行 SQL，返回查询结果或错误信息
4. data_interpretation_node：调用 LLM 对查询结果做业务解读，同时基于规则生成 ECharts 图表
5. error_handling_node：重试超限后返回友好错误提示

流程：用户输入 → schema_rag → intent_and_sql → (如果是闲聊直接 END) → sql_execute → (成功 → data_interpretation → END；失败 → 重试最多3次 → error_handling → END)

Q4: 你提到 LLM 调用次数减少了 66%，具体怎么做的？
A4: v1.0 有 3 次 LLM 调用：router_node（意图识别）、text_to_sql_node（SQL 生成）、data_interpretation_node（业务解读）。
v1.1 把 router_node 和 text_to_sql_node 合并为 intent_and_sql_node，一次 LLM 调用同时完成意图识别和 SQL 生成，通过 JSON Mode 强制输出结构化结果。这样数据查询场景从 3 次 LLM 调用降到 2 次（intent_and_sql + data_interpretation），闲聊场景只需 1 次。整体平均减少约 66%。

Q5: 什么是 JSON Mode？为什么要用它？
A5: JSON Mode 是在调用 LLM 时设置 response_format={"type": "json_object"}，强制模型输出合法的 JSON 格式。
好处是：
- 不需要用正则去解析模型的自由文本输出，避免解析失败
- 输出结构可控，比如我要求输出 {"intent": "data_query", "sql": "SELECT ..."}
- 减少后处理代码复杂度，提高稳定性

Q6: SQL 安全机制是怎么做的？
A6: 三层防护：
1. Prompt 层面：在系统提示词中明确要求"仅允许 SELECT 语句，禁止 INSERT/UPDATE/DELETE/DROP 等"
2. 代码校验层面：sql_executor.py 中的 validate_sql_safety 函数，用正则检查 SQL 是否以 SELECT 开头，并拦截所有危险关键词（INSERT, UPDATE, DELETE, DROP, ALTER, CREATE, TRUNCATE）
3. 自动 LIMIT：所有查询自动添加 LIMIT 100，防止全表扫描导致性能问题

Q7: SQL 执行失败了怎么办？
A7: 我实现了 SQL 自愈机制。sql_execute_node 执行失败后，通过条件边检查 retry_count：
- 如果 retry_count < 3，将 error_message 写入状态，回到 intent_and_sql_node，LLM 会根据错误信息（比如"字段不存在""语法错误"）重新生成修正后的 SQL
- 如果重试 3 次仍然失败，进入 error_handling_node 返回友好错误提示
error_message 会作为 Prompt 的一部分传给 LLM，让模型能针对性修正。

Q8: 图表是怎么生成的？
A8: 当前版本采用规则匹配而非 LLM 生成：
- 双列数据且 ≤10 行 → 饼图（pie），自动提取 name/value
- 单列数据且 ≤20 行 → 柱状图（bar）
- 其他情况 → 不生成图表
配色采用莫兰迪色系，背景透明，隐藏多余网格线。
虽然写了 ECHARTS_CONFIG_SYSTEM_PROMPT 用于 LLM 生成图表，但为了减少调用次数和延迟，目前没启用。

Q9: 前后端是怎么通信的？
A9: 后端用 FastAPI 提供 RESTful API（POST /api/chat），前端 Streamlit 通过 httpx 发送 POST 请求。
请求体包含 message 和可选的 conversation_id，响应体返回 final_answer（业务解读）、echarts_config（图表配置）、以及 metadata（SQL、执行耗时、token 消耗等）。
后端还配置了 CORS 中间件和全局耗时中间件，响应头中包含 X-Process-Time。

Q10: 这个项目有什么已知的不足或未来优化方向？
A10: 主要有几点：
1. Schema RAG 未实现向量检索：当前硬编码完整 Schema，表多时 Prompt 会很长，应该用 ChromaDB 做向量化检索 Top-K 相关表
2. 图表生成比较简单：只有饼图和柱状图，可以启用 LLM 生成更丰富的图表类型
3. 不支持多轮对话：每次查询独立，没有上下文记忆
4. 流式响应是伪流式：可以改为真正的分步流式，先返回"正在生成 SQL..."再返回结果

二、基于大模型与多模态特征融合的算法研究
########################################

Q11: 你的研究课题是什么？目前做到什么阶段？
A11: 研究方向是基于大模型与多模态特征融合的算法，目前还在探索阶段。已完成的工作包括：
- 跑通了 ESM2（3B 参数蛋白质语言模型）的加载、显存优化和下游任务微调流程
- 探索了多模态模型架构设计，实现了注意力池化等自定义模块
- 建立了一套基于运行时日志的 Debug 工作流

Q12: ESM2 是什么模型？你怎么做的显存优化？
A12: ESM2 是 Meta AI 发布的蛋白质语言模型，基于 Transformer 架构，3B 参数量。
显存优化的常见手段包括：
- 混合精度训练（FP16/BF16）
- 梯度累积（gradient accumulation）减少单步 batch size
- 梯度检查点（gradient checkpointing）用计算换显存
- 冻结部分层只微调顶层
具体哪些方法有效需要根据实际显存限制和任务需求来选择。

Q13: 什么是注意力池化？
A13: 注意力池化是一种将变长序列聚合为固定维度表示的方法。不同于简单的平均池化或最大池化，注意力池化会学习一组权重，对序列中不同位置赋予不同重要性，然后加权求和。这样可以让模型关注序列中信息量更大的部分，在多模态融合中常用于将不同模态的特征对齐到统一表示空间。

Q14: 你说你会用 AI 辅助定位工程问题，能举个具体例子吗？
A14: 比如在加载 ESM2 时遇到 CUDA out of memory 错误，我会把完整的 traceback 和当前代码发给 AI，让它分析具体是哪一步导致显存溢出，然后建议用梯度检查点或者降低 batch size。
还有一次遇到 PyTorch 和 transformers 库版本冲突的问题，报了一个很隐晦的 AttributeError，通过 AI 分析日志快速定位到是版本不兼容，升级后解决。

Q16: 百度 AI 图像识别集成是怎么做的？
A16: 整体链路是：前端文件上传 → 后端接收图片 → 转为 Base64 编码 → 调用百度 AI 开放平台图像识别 API → 解析返回的 JSON 结果 → 前端渲染识别结果。
需要处理的细节包括：图片格式校验、Base64 编解码、API 鉴权（access_token 获取与刷新）、错误重试等。

四、专业技能相关
########################################

Q17: 解释一下 Transformer 的核心机制？
A17: Transformer 的核心是自注意力机制（Self-Attention）：
- 每个 token 通过线性变换得到 Query、Key、Value 三个向量
- 通过 Q 和 K 的点积计算注意力权重（除以 sqrt(d_k) 防止梯度消失），再经过 Softmax 归一化
- 用权重对 V 加权求和，得到每个位置的输出
- 多头注意力（Multi-Head Attention）让模型从不同子空间捕获不同类型的依赖关系
此外还有残差连接、Layer Normalization、前馈网络（FFN）和位置编码。

Q18: 什么是 RAG？和直接用 LLM 有什么区别？
A18: RAG（Retrieval-Augmented Generation）是"检索增强生成"：
1. 先用用户问题检索相关文档片段（通常是向量相似度检索）
2. 把检索到的内容拼接到 Prompt 中作为上下文
3. LLM 基于这些上下文生成回答
和直接用 LLM 的区别：
- 直接用 LLM：只依赖模型训练时学到的知识，容易产生幻觉，无法回答私有数据
- RAG：引入外部知识库，回答更准确、可追溯，适合企业私有数据问答场景

Q19: 什么是 Text-to-SQL？有哪些难点？
A19: Text-to-SQL 是将用户的自然语言问题转换为可执行的 SQL 查询。
难点包括：
- 表和字段的映射：用户说"销售额"，模型需要知道对应 orders 表的 amount 字段
- 复杂查询：多表 JOIN、子查询、嵌套聚合、时间函数等
- 歧义消解：用户表达模糊时需要合理推断意图
- 安全性：防止生成危险 SQL（注入攻击、数据篡改）
- 方言差异：不同数据库（MySQL/PostgreSQL/SQLite）的 SQL 语法有差异

Q20: Prompt Engineering 有哪些常用技巧？
A20: 常用技巧包括：
- 角色设定：给模型一个明确身份（"你是一个专业的数据分析师"）
- 结构化指令：用 Markdown 标题、列表、代码块组织 Prompt，提高可读性
- Few-shot 示例：提供输入输出样例，让模型模仿格式
- 输出约束：指定输出格式（JSON Mode、固定字段名），减少后处理
- 链式思考（CoT）：让模型先分析再回答，提升推理质量
- 负面约束：明确说"不要做什么"，比如"禁止生成 DELETE 语句"

Q21: FastAPI 和 Flask 有什么区别？为什么选 FastAPI？
A21: 主要区别：
- FastAPI 原生支持异步（async/await），性能更高
- FastAPI 基于 Pydantic 做请求/响应模型验证，类型安全
- FastAPI 自动生成 Swagger/OpenAPI 文档，方便调试
- Flask 更轻量、生态更成熟，但需要手动处理异步和参数校验
选 FastAPI 是因为项目需要对接 LLM API（涉及网络 IO），异步支持很重要，同时 Pydantic 的自动校验减少了样板代码。

Q22: 你平时怎么使用 Windsurf 和 Claude Code CLI 这些 AI 编程工具的？
A22: 
- Windsurf：主要用来做项目级别的代码生成和重构，它能理解整个项目上下文，帮我快速搭建项目骨架、编写模块代码、调整排版样式等
- Claude Code CLI：在终端里快速提问和生成代码片段，适合不想离开终端的场景
- 日常还会用 Gemini、ChatGPT 做技术方案调研、错误排查、代码 review
核心思路是：自己先理解需求和架构设计，用 AI 加速执行层面的工作，最终代码自己要能完全理解和维护。

五、场景题 / 开放题
########################################

Q23: 如果你的 DataInsight Agent 要支持 100 张表的大型数据库，你会怎么改？
A23: 核心是把 schema_rag_node 从硬编码改为向量检索：
1. 预处理阶段：把每张表的 DDL + COMMENT 做 embedding，存入向量数据库（如 ChromaDB）
2. 查询阶段：对用户问题做 embedding，检索 Top-K 最相关的表结构
3. 只把相关表的 Schema 放入 Prompt，大幅减少 token 消耗
4. 可以再加一层 Schema 缓存，对高频查询模式复用检索结果

Q24: 如果用户连续追问（多轮对话），你会怎么实现？
A24: 需要做几件事：
1. 后端维护一个 conversation_id → chat_history 的映射（可以用内存字典或 Redis）
2. 每次请求时把历史对话传入 intent_and_sql_node 的 Prompt
3. LLM 可以根据上下文理解指代关系，比如"那上个月呢"指的是之前查询的指标
4. 注意控制历史长度，避免 token 超限（可以只保留最近 N 轮）

Q25: 如果面试官说"你这个项目看起来比较简单"，你怎么回应？
A25: 可以说：
"这个项目的核心价值不在于代码量，而在于我完整走通了一个 LLM Agent 应用的全链路：从 Prompt 设计、状态机编排、SQL 安全机制、错误重试，到前后端联调和性能优化。在这个过程中，我理解了 Agent 工作流的核心模式，也积累了 Text-to-SQL、JSON Mode、LangGraph 状态管理等实践经验。同时我也清楚它的不足——比如 Schema RAG 未实现向量检索、不支持多轮对话——这些是我接下来要改进的方向。"

Q26: 你在项目中遇到过最大的困难是什么？怎么解决的？
A26: DataInsight Agent 中最大的困难是 v1.0 响应时间太长（5-15 秒），用户体验差。
分析后发现是 3 次串行 LLM 调用导致的。我的解决方案是合并意图识别和 SQL 生成为一次调用，使用 JSON Mode 保证输出格式。改完后 LLM 调用次数从 3 次降到 1-2 次，响应时间降到 4-8 秒。
这个经验让我理解到：LLM 应用的性能瓶颈通常在模型调用次数上，应该尽量合并能一次完成的任务。

Q27: 你怎么看待 AI 辅助编程？它的边界在哪里？
A27: AI 编程工具是非常强大的效率放大器，但不能替代开发者的判断力：
- 适合的场景：样板代码生成、API 查询、错误排查、代码重构、文档编写
- 需要人来做的：架构设计、需求理解、安全审查、最终代码质量把关
- 关键原则：AI 生成的代码自己必须能完全理解和维护，不能当黑盒用
我的做法是先自己想清楚技术方案，再用 AI 加速实现，最后自己 review。

Q28: 你为什么想做大模型应用开发方向？
A28: 两个原因：
1. 技术趋势：大模型正在重新定义软件开发的范式，从传统的"写规则"变成"设计 Prompt + 编排 Agent"，我认为这是未来几年最有价值的工程方向之一
2. 个人兴趣：我在做 DataInsight Agent 的过程中，切实体会到了 LLM + Agent 能让非技术用户用自然语言操作数据，这种"降低技术门槛"的价值让我很有成就感
